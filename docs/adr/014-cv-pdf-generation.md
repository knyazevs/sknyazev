# ADR-14: Генерация CV в формате PDF во время сборки

- Статус: Предложено
- Дата: 2026-04-13
- Авторы: Sergey Knyazev, Claude Sonnet 4.6
- Теги: cv, build, pdf, генерация, контент

## Контекст

Проект содержит структурированный профессиональный контент в `docs/`:
опыт работы, навыки, образование, профессиональный профиль.

Ранее существовал `CV.md` — ручная копия тех же данных.
Ручная копия нарушает ADR-2 (единый источник истины):
при любом обновлении `docs/` CV расходился с реальностью.
`CV.md` удалён.

Цель: CV PDF должен генерироваться автоматически из `docs/` при каждом build,
всегда оставаясь актуальным без ручного вмешательства.

---

## Решение

### CV PDF — слой представления (ADR-2)

CV PDF является производным от источника истины.
Обратного потока не существует: PDF не редактируется вручную.

```
docs/ (Source of Truth)
        ↓
  cv-builder script
        ↓
  HTML template
        ↓
  app/dist/cv.pdf
```

### Источники данных для сборки CV

| Данные | Источник |
|---|---|
| Имя, контакты, локация | `docs/profile/index.md` (front matter) |
| Профессиональное summary | `docs/profile/index.md` (body) |
| Образование | `docs/profile/04-education.md` |
| Опыт работы | `docs/experience/*.md` |
| Навыки | `docs/skills/*.md` |

Отдельного CV-конфига нет. Все данные живут в обычных профильных файлах
и используются как сайтом, так и CV builder.

### Расширение front matter файлов `experience/`

Текущий `author-role` — семантический тег для RAG, не должность для CV.
Текущие company-названия присутствуют только в H1-заголовке файла.

Добавить в front matter три поля:

```yaml
---
type: experience
# ... существующие поля без изменений ...
company: "Название компании"      # отображаемое название для CV
title: "Tech Lead / Architect"    # должность для CV
cv_description: "2–3 предложения" # краткое описание роли для CV
---
```

`company` и `cv_description` обязательны для файлов с коммерческим опытом.
Для experience-файлов без явного работодателя (фриланс) `company` не указывается.

### Расширение front matter `docs/profile/index.md`

Добавить персональные и контактные данные в front matter:

```yaml
---
type: profile
name: Сергей Князев
title: Technical Lead / Architect
birth_year: 1996
location: Россия
contacts:
  github: ""
  linkedin: ""
  telegram: ""
  email: ""
---
```

Эти данные используются и сайтом, и CV builder — без дублирования.

### CV builder — `scripts/build-cv.ts`

Bun TypeScript-скрипт:
1. Читает front matter и body всех источников
2. Собирает структурированный CV-объект
3. Рендерит HTML через шаблон `scripts/cv-template.html`
4. Сохраняет PDF через Puppeteer → `app/dist/cv.pdf`

### Интеграция в build

```json
"scripts": {
  "build": "bun run build:cv && bun run build:app",
  "build:cv": "bun scripts/build-cv.ts"
}
```

GitHub Actions не меняется: `bun run build` уже покрывает оба шага.

### Технология генерации PDF — Puppeteer

Альтернативы: Pandoc, Typst, LaTeX.

Puppeteer выбран по трём причинам:
- npm-зависимость без системных бинарей в CI
- верстка через CSS — нет отдельного языка шаблонов
- `ubuntu-latest` поддерживает headless Chrome из коробки

---

## Последствия

### Положительные
- CV всегда актуален: любой push пересобирает PDF
- Нет дублирования данных: один источник для сайта, LLM и CV
- CV доступен как статический файл на сайте (`/cv.pdf`)

### Отрицательные / Стоимость
- Puppeteer добавляет ~150 MB к `node_modules` (Chromium)
- Нужно поддерживать HTML-шаблон при смене дизайна
- `company` и `title` обязательны в experience-файлах — требует дисциплины

---

## Рассмотренные альтернативы

**Pandoc** — внешняя системная зависимость, сложнее контролировать
кириллицу в CI.

**Typst** — современный, хороший кандидат, меньше npm-экосистема.
Может быть пересмотрен.

**LaTeX** — избыточная сложность для CV, системная зависимость.
