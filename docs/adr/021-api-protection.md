# ADR-21: Защита API от злоупотреблений

- Статус: Принято
- Дата: 2026-04-14
- Авторы: Sergey Knyazev, Claude Sonnet 4.6
- Теги: security, rate-limit, llm, api, защита

## Контекст

Бэкенд (ADR-19) проксирует платные AI-сервисы: OpenRouter (LLM), OpenAI TTS и Whisper.
Каждый запрос к `/api/chat` стоит реальных денег.

Без защиты возможны два сценария злоупотребления:

1. **Нецелевые запросы** — чатбот используется не по назначению
   (генерация произвольного контента, code generation, общие вопросы).
   Нарушает ADR-3 (LLM как строго ограниченный интерпретатор) и создаёт финансовые потери.

2. **Rate abuse** — намеренный или случайный flood запросов от одного IP
   (скрипт, краулер, повторные нажатия), который исчерпывает лимиты OpenRouter/OpenAI.

---

## Решение

Два независимых механизма защиты.

### 1. LLM-классификатор релевантности вопроса

Перед запуском RAG-пайплайна выполняется быстрый бинарный вызов к дешёвой модели
(`openai/gpt-4o-mini` или аналог через OpenRouter):

```
System: Ты — фильтр релевантности. Отвечай строго YES или NO, без объяснений.
User: Этот вопрос о конкретном человеке, его профессиональном опыте, навыках,
проектах или архитектурных решениях? Вопрос: "{question}"
```

- `YES` → запрос передаётся в RAG-пайплайн
- `NO` → возвращается `400 Bad Request` с объяснением
- Ошибка классификатора → **fail open** (запрос пропускается)
  _Обоснование: сбой защиты не должен ломать основной пользовательский сценарий_

Модель классификатора — отдельный конфигурируемый параметр (`CLASSIFIER_MODEL`),
не связанный с основным LLM. Стоимость ~$0.0001 на запрос.

### 2. Rate Limiting (Ktor встроенный плагин)

Используется `io.ktor:ktor-server-rate-limit` — нативный плагин Ktor,
реализующий фиксированное окно по ключу (IP-адрес запроса).

Два уровня:

| Группа | Маршруты | Лимит |
|--------|----------|-------|
| `chat` | `/api/chat` | 10 req / мин |
| `media` | `/api/tts`, `/api/asr` | 20 req / мин |

При превышении → `429 Too Many Requests` (стандартный HTTP-ответ).

Состояние хранится in-memory, сбрасывается при рестарте сервера.
Для CV-портфолио с единственным экземпляром сервиса — достаточно.

## Обоснование

### Почему LLM-классификатор, а не keyword-filter

Keyword-фильтр (regex/список слов) хрупок:
- русскоязычные парафразы, опечатки, косвенные формулировки
- ложные срабатывания на вопросы об архитектуре (например, «как это устроено?»)

LLM-классификатор понимает намерение, а не ключевые слова.
Стоимость одного вызова `gpt-4o-mini` пренебрежимо мала по сравнению
со стоимостью полного RAG-запроса.

### Почему Ktor RateLimit, а не Redis

Redis добавляет операционную сложность (отдельный сервис, конфигурация, бэкапы)
без выгоды для single-instance деплоя. Ktor-плагин решает задачу без зависимостей.

### Почему fail open для классификатора

Если API классификатора недоступен — лучше пропустить запрос, чем сломать
основной пользовательский сценарий (посетитель не может задать вопрос о проекте).
Потери в этом случае минимальны (сбой обычно кратковременен).

## Альтернативы и причины отказа

### Cosine similarity к профильному вектору
**Отказано**: порог эмпирический, плохо работает на коротких/абстрактных вопросах,
требует ручной настройки при изменении корпуса документов.

### API-ключ для клиента
**Отказано**: ключ в клиентском JS легко извлечь. Не решает проблему abuse.

### Redis sliding window rate limit
**Отказано на старте**: избыточно для single-instance деплоя портфолио.
Абстракция слоя rate limit позволяет добавить Redis при необходимости.

## Последствия

### Положительные
- Финансовые риски ограничены
- ADR-3 технически обеспечен на уровне инфраструктуры
- 429 сигнализирует клиенту о превышении лимита — UX-понятно

### Отрицательные / Стоимость
- Дополнительный API-вызов на каждый чат-запрос (+~100ms latency)
- In-memory rate limit не персистентен между рестартами
- Классификатор может давать ложные срабатывания на граничные вопросы

## Связь с другими ADR

- ADR-19 — бэкенд Kotlin/Ktor: плагин нативен для Ktor
