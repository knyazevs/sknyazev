# ADR-20: Архитектура RAG-пайплайна

- Статус: Принято
- Дата: 2026-04-14
- Авторы: Sergey Knyazev, Claude Sonnet 4.6
- Теги: rag, llm, embeddings, vector-store, архитектура

## Контекст

ADR-3 определяет LLM как **интерактивный интерпретационный слой**,
работающий строго поверх источника истины (документации репозитория).

LLM без контекста документов:
- генерирует правдоподобные, но неверные утверждения об авторе
- размывает границу между задокументированным опытом и «дополнениями»
- снижает доверие к проекту как инженерному артефакту

Для выполнения требований ADR-3 нужен механизм,
который передаёт LLM только релевантные фрагменты документации,
а не весь корпус (слишком велик для контекста) и не ничего.

---

## Решение

Реализовать **RAG (Retrieval-Augmented Generation) пайплайн**
как часть Kotlin/Ktor бэкенда (ADR-19).

### Схема пайплайна

```
Запрос пользователя
       │
       ▼
  [EmbeddingService]
  OpenAI text-embedding-3-small
       │
       ▼
  [InMemoryVectorStore]
  cosine similarity → top-5 чанков
       │
       ▼
  [RagPipeline]
  system prompt (ADR-3 constraints) + chunks
       │
       ▼
  [OpenRouterClient]
  LLM streaming (SSE)
       │
       ▼
  Ответ + источники
```

### Инициализация при старте

1. `DocumentLoader` обходит `docs/` рекурсивно, читает `.md` файлы
2. Каждый файл разбивается на чанки по заголовкам H2/H3
   (fallback — скользящее окно ~512 токенов с перекрытием 64)
3. Каждый чанк эмбеддируется через `EmbeddingService`
4. Вектор + метадата (путь к файлу, заголовок секции) → `InMemoryVectorStore`

### Векторное хранилище

```kotlin
interface VectorStore {
    suspend fun index(entry: VectorEntry)
    suspend fun search(queryVector: FloatArray, topK: Int): List<VectorEntry>
}

data class VectorEntry(
    val id: String,
    val content: String,
    val filePath: String,
    val sectionTitle: String,
    val vector: FloatArray
)
```

`InMemoryVectorStore`: хранит `List<VectorEntry>`, поиск — полный перебор
с cosine similarity. При корпусе ~20 файлов / ~100 чанков производительность достаточна.

### System prompt (ADR-3)

```
Ты — интерпретационный слой персонального профиля Сергея Князева.
Отвечай ТОЛЬКО на основе предоставленных фрагментов документации.
Всегда указывай источник (путь к файлу / название ADR).
Если информации недостаточно — явно скажи об этом.
Не добавляй факты, которых нет в источнике.
```

### Эмбеддинг-модель

**OpenAI `text-embedding-3-small`**:
- размерность 1536, достаточна для задачи
- низкая стоимость (дешевле `text-embedding-3-large`)
- доступна напрямую через OpenAI API

## Альтернативы и причины отказа

### Внешняя векторная БД (pgvector, Chroma, Weaviate)
**Отказано на старте**: docs/ содержит ~20 файлов / ~100 чанков.
Внешняя БД добавляет операционную сложность (отдельный сервис, миграции,
бэкапы) без выигрыша в производительности для данного объёма.

Абстракция `VectorStore` сохраняет возможность миграции на pgvector
без изменения бизнес-логики.

### Передача всей документации в контекст LLM
**Отказано**: общий объём docs/ превышает разумный размер контекста
для дешёвых моделей. Кроме того, передача нерелевантных документов
увеличивает шум и стоимость запроса.

### Keyword-based поиск (BM25, TF-IDF)
**Отказано**: семантический поиск через эмбеддинги лучше справляется
с парафразами и русскоязычными запросами, где вопрос и ответ
сформулированы по-разному.

### LlamaIndex / LangChain (JVM-порт)
**Отказано**: зрелость JVM-портов ниже, добавляет непрозрачную зависимость.
Самописный пайплайн (~200 строк Kotlin) полностью контролируем
и демонстрирует понимание механики RAG.

## Последствия

### Положительные
- LLM работает строго в рамках ADR-3 — только по источнику истины
- Ответы трассируемы к конкретным файлам документации
- Нет внешних зависимостей (только JVM + OpenAI API)
- Абстракция `VectorStore` позволяет мигрировать без переработки логики

### Отрицательные / Стоимость
- Индекс в памяти: при рестарте сервера эмбеддинги пересчитываются
  (~секунды при текущем объёме, ~доли секунды после кэширования файлов OS)
- Полный перебор O(n) при поиске: допустимо до ~10000 чанков
- Зависимость от OpenAI Embeddings API (стоимость и uptime)
