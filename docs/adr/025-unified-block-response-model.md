# ADR-25: Единая блочная модель ответа агента

- Статус: Принято
- Дата: 2026-04-22
- Авторы: Sergey Knyazev, Claude Opus 4.7
- Теги: ui, blocks, sse, streaming, frontend, architecture

## Контекст

ADR-024 ввёл UI-блоки (image, image_gallery, link_list, text_with_image) как **дополнение** к основному потоку токенов. На практике это создаёт UX-проблему: блоки визуально существуют отдельно от ответа — пристёгнутыми «виджетами» в зоне `.msg-blocks` под текстом. Пользователь воспринимает их как инородный элемент, а не как часть ответа.

Корень проблемы — в самой модели данных. Ответ сейчас состоит из **двух параллельных сущностей**: потока токенов (основное тело) и массива блоков (дополнения). Визуально это всегда будет раздельно, какие бы стили мы ни добавили. Попытка сделать inline-блоки через offset-позиционирование — костыль поверх неподходящей модели.

Правильное решение: ответ — это **единый упорядоченный массив блоков**, где обычный текст — один из типов блоков (`text`), наравне с `image`, `link_list` и т.д. Тогда LLM может естественно смешивать текстовые секции с визуальными: абзац → картинка → ещё абзац → ссылки → карточка. Inline становится не спецфункцией, а следствием модели.

## Решение

### Каталог типов блоков

Добавляется новый тип:

| Тип | Поля | Стриминг | Назначение |
|-----|------|----------|------------|
| `text` | `{content: String}` — markdown | **Да** (по токенам) | Абзац или секция обычного текста |
| `image` | см. ADR-024 | Нет | Без изменений |
| `image_gallery` | см. ADR-024 | Нет | Без изменений |
| `link_list` | см. ADR-024 | Нет | Без изменений |
| `text_with_image` | см. ADR-024 | Нет | Без изменений |

### Как появляются text-блоки

LLM не вызывает `render_text` явно (добавлять псевдо-tool избыточно). Бэкенд **автоматически оборачивает** токен-поток в text-блоки по правилу:

- Первый эмитнутый токен → `block_start {type: "text", id: Tn}`, далее токены эмитятся как `token {id: Tn, delta: "..."}`.
- Приход `tool_call` → `block_end {id: Tn}`, исполнение tool → `block {id: Xn, type: "image"|..., data}`, следующий токен (если есть) → новый `block_start {type: "text", id: Tn+1}`.
- Конец ответа (`[DONE]`) → `block_end` текущего text-блока, `[DONE]`.

Правило даёт: последовательный ответ без tool-calls = один text-блок. Ответ с tool-calls = чередование `text / tool-блоки / text`. Это естественное inline-поведение без специальных механизмов позиционирования.

### SSE-протокол

К существующим событиям (`sources`, `suggestions`, `error`, `[DONE]`) добавляются:

```
data: {"block_start": {"id": "T1", "type": "text"}}
data: {"token": {"id": "T1", "delta": "Здрав"}}
data: {"token": {"id": "T1", "delta": "ствуй, я..."}}
data: {"block_end": {"id": "T1"}}
data: {"block": {"id": "B1", "type": "image_gallery", "data": {...}}}
data: {"block_start": {"id": "T2", "type": "text"}}
data: {"token": {"id": "T2", "delta": "..."}}
data: {"block_end": {"id": "T2"}}
```

Старый формат событий `{"token": "..."}` (без id) **удаляется** — это breaking change для SSE-потребителей. Наш единственный клиент — фронт — обновляется в том же коммите.

### Формат Message на фронте

```typescript
interface Message {
    question: string;
    blocks: Block[];
    sources: string[];            // оставляем отдельно — они про цитирование [1]
    // answer, detail — удаляются
}

type Block =
    | { id: string; type: "text"; content: string; streaming?: boolean }
    | { id: string; type: "image" | "image_gallery" | "link_list" | "text_with_image"; ...ADR-024 поля }
```

### Маркер `---DETAIL---`

Сохраняется **внутри** text-блока. Если text-блок содержит маркер, фронт разбивает его на summary + detail визуально (так же как сейчас), без разделения на два блока на уровне данных. Это сохраняет существующее поведение «Подробнее» без усложнения модели.

Если LLM решит выдать summary и detail **в разных** text-блоках (разделённые tool-call'ом) — фронт их показывает как есть, `---DETAIL---` игнорируется если не встречается в одном блоке. Это edge-case, допустимо.

### TTS

TTS подписывается на токены text-блоков с детекцией `---DETAIL---` (stop-условие). Логика остаётся прежней, меняется только источник: вместо единого `token`-потока — токены конкретного text-блока (первого с маркером summary, либо всего блока если маркера нет).

### Миграция истории

sessionStorage `HISTORY_KEY` содержит массив Message старой формы `{question, answer, sources, blocks}`. При загрузке:

```typescript
function migrateLegacyMessage(m: any): Message {
    if (Array.isArray(m.blocks) && typeof m.answer === "string") {
        // старая форма
        const textBlock: Block = { id: "legacy-text", type: "text", content: m.answer };
        // старые blocks (image/link_list/...) сохраняются и идут после текста
        return { question: m.question, sources: m.sources ?? [], blocks: [textBlock, ...m.blocks] };
    }
    return m;  // уже новая форма
}
```

Потеря информации: нельзя восстановить inline-позиции визуальных блоков в старых сессиях — они встают после текста, как было. Это ок — старые сессии не перегенерируются.

### Санитайзер markdown

Применяется к `content` text-блока при рендеринге — удаляет `![](...)` и внешние `[text](url)` как сейчас. Логика из ADR-024 не меняется.

### Рендеринг на фронте

Единый цикл `{#each msg.blocks as b}` — каждый блок через `BlockRegistry`. Для text-блока — новый компонент `TextBlock.svelte` с marked + санитайзер + детекция `---DETAIL---` + «Подробнее». Зоны `.msg-answer` и `.msg-blocks` удаляются как отдельные сущности — всё в `.msg-body`.

### Цитирование [1]

Cite-badge (`[1]` → `<button class="cite-badge">`) применяется к markdown-рендеру внутри text-блока. Логика и CSS переносятся в TextBlock.

## Обоснование

- **Модель соответствует природе ответа.** LLM генерирует линейный ответ с вкраплениями визуальных элементов — это и есть последовательность блоков.
- **Inline — бесплатно.** Не нужен offset-позиционирование, маркеры, или любые другие костыли. LLM вызвала tool между абзацами → блок встаёт между text-блоками.
- **Тестируемо и сериализуемо.** История сообщений — массив блоков, тривиально диффить, логировать, восстанавливать.
- **Расширяемо.** Новые типы блоков (таблицы, диаграммы, embed) регистрируются без изменения ядра — так же как сейчас.
- **Убирает двойную систему.** Была `answer: string` + `blocks: UiBlock[]`, стало `blocks: Block[]`. Меньше состояния, меньше багов рассинхрона.

## Последствия

### Положительные
- Цельный визуальный ответ без «пристёгнутых» виджетов.
- Inline-блоки работают из коробки.
- Единая модель рендеринга на фронте — проще добавлять новые типы.
- История сохраняется с полной структурой, не теряет визуального порядка.

### Отрицательные / Компромиссы
- SSE-протокол breaking change. Любой внешний потребитель SSE потребует обновления. Сейчас у нас один фронт — риск управляемый.
- Бэкенд ChatRoutes усложняется state-машиной (открытый text-блок, переключение по tool-calls).
- TTS подписывается на поток токенов конкретного блока — небольшая переделка.
- sessionStorage-миграция одноразовая, но нужна — иначе старые сессии ломаются.
- Санитайзер применяется per-блок, не глобально — требует аккуратного порта.

## Рассмотренные альтернативы

### Вариант A — только стилистика (убрать рамки у блоков)
Отклонено пользователем: не решает корня проблемы. Блоки остаются отдельной сущностью после текста.

### Вариант Б — inline через offset
Отклонено: костыль поверх двойной модели. Требует бэкенду считать offset и фронту вставлять блоки в HTML — хрупко, усложняет рендер, не масштабируется на сложные вёрстки.

### Добавить отдельный tool `render_text`
Отклонено: LLM пришлось бы постоянно оборачивать текст в tool-call, что удваивает стоимость токенов и нагрузку на модель. Бэкенд-оборачивание токенов дешевле и надёжнее.

### Оставить `answer: string` + blocks, но рендерить их в одном контейнере
Полу-мера: данные всё ещё двойные, нельзя естественно переставлять. Быстрое решение, но не решает «инородности» на уровне модели. При первой же просьбе «картинку в середину» упрёмся.

## Связь с другими ADR

- **ADR-024** (UI-блоки) — базовый каталог и C/B-механизмы. ADR-025 меняет способ **сборки** блоков в ответ, не трогает сам каталог и механизмы извлечения.
- **ADR-023** (LLM fallback) — инвариант «fallback только до первого emit» сохраняется. Первый emit теперь — либо Token, либо Block, либо ToolCall — любой из них закрывает окно fallback.
- **ADR-020** (unified content modal) — не затронут, модалка работает с источниками по-прежнему.
