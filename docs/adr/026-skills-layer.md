# ADR-26: Skills — декларативный слой инструкций поверх RAG

- Статус: Принято
- Дата: 2026-04-22
- Авторы: Sergey Knyazev, Claude Opus 4.7
- Теги: rag, agent, routing, prompts, architecture

## Контекст

RAG-пайплайн (ADR-016, ADR-019) оптимизирован под «точечные» вопросы: HyDE → dense + BM25 → RRF → cosine-filter выбирает top-5 чанков, релевантных конкретному запросу. На этом хорошо работают вопросы вида «какой стек у платформы МРС?» или «что сказано в ADR-019?».

Где он ломается — **обзорные запросы**: «какие проекты делал Сергей?», «расскажи о всём опыте», «какие ADR приняты?». У нас 8 файлов в `docs/experience/`, 9 в `docs/projects/`, 25 ADR. Top-5 чанков не дают полной картины: модель честно отвечает по тем пяти, что попались, но обзора не получается. Пользователь видит частичный ответ и не понимает, что упустил.

Две системные причины:
1. **Retrieval выбирает best-of-N, а не all-of-N.** Даже если поднять top_K до 20, это не решает обзорные кейсы: во-первых, не влезет в контекст; во-вторых, для настоящего «список всех проектов» нужна не выборка, а **обход директории**.
2. **System prompt у нас один на все вопросы** — общий интерпретационный. Он хорош для точечных ответов, но для обзоров нужны другие инструкции: «структурируй по разделам», «для каждого проекта — имя/роль/стек/задача», «не пропускай ни один файл».

Нужен механизм, позволяющий распознать класс вопроса и переключиться на **специализированный режим ответа**: другой retrieval (обход директории целиком), другой system prompt (инструкция под конкретный класс запросов). По смыслу это то же, что Claude Skills — декларативный файл с триггерами и инструкцией, который активируется при совпадении.

## Решение

### Формат skill-файла

Skills живут в `server/skills/*.md` — рядом с `server/src/`, **не** в `docs/`. Это конфигурация агента, а не контент проекта. Грузятся okio-шным FileSystem так же, как документы.

```markdown
---
name: projects-overview
description: Полный обзор всех проектов (коммерческих и личных)
triggers: [какие проекты, над чем работал, список проектов]
sources:
  - docs/experience/
  - docs/projects/
---
# Инструкция агенту

Дай полный структурированный обзор. Раздели на «Коммерческий опыт» и
«Личные проекты». Для каждого — название, роль, стек, 1-2 предложения
о задаче. Не пропускай ни один файл.
```

- `name` — уникальный идентификатор, используется в роутере
- `description` — одно предложение, показывается LLM-классификатору
- `triggers` — список фраз-примеров для промпта классификатора (не regex, LLM сама обобщает)
- `sources` — список путей: директория (загрузить всё `.md` рекурсивно) или одиночный файл. Пути относительно рабочей директории процесса (совместимы с `docsPath`)
- Тело после frontmatter — **system prompt** конкретного skill'а

### Роутер = расширенный guard

Вместо двух LLM-вызовов (guard + router) объединяем в один. `QuestionGuard` переименовывается в `QuestionRouter` и возвращает:

```kotlin
sealed class RoutingDecision {
    object Irrelevant : RoutingDecision()
    object GenericRag : RoutingDecision()
    data class UseSkill(val name: String) : RoutingDecision()
}
```

Промпт формируется динамически из списка загруженных skills:

```
Classify the user question into one of:
- IRRELEVANT — not about Sergey or his project
- SKILL:<name> — matches one of the listed skills below
- GENERIC — about Sergey but none of the skills fit

Available skills:
- projects-overview: Полный обзор всех проектов. Triggers: какие проекты, над чем работал...
- skills-overview: Навыки и стек. Triggers: какой стек, какие технологии...
...

Return a single line: IRRELEVANT, GENERIC, or SKILL:<name>.
```

Circuit breaker сохраняется: при tripped → `Irrelevant` (как сейчас fail-closed). При транзиентной ошибке классификатора — fail-open в `GenericRag` (пользователь не блокируется, просто не получает skill-обработки).

### SkillExecutor вместо retrieval

Для `UseSkill(name)` идёт **альтернативный код-путь**:

1. Резолвим `skill.sources` в список файлов (рекурсивный обход директорий)
2. Читаем файлы целиком, оцениваем размер контекста (грубо: `chars / 4`)
3. Если > 100K токенов — бросаем исключение, `ChatRoutes` конвертирует в 500. Фоллбека в generic RAG не делаем (осознанное решение: skill сломан → чинить skill, а не маскировать проблему)
4. Извлекаем UI-блоки через `BlockExtractor.extract` пофайлово, агрегируем, cap 5
5. Строим prompt: system = skill.instruction + общие цитирования/формат, user = контекст с `[1] path\ncontent\n---\n[2] path\ncontent` + оригинальный вопрос
6. Стримим через `openRouterClient.streamChat` — возвращаем `RagResult` того же типа, что и `RagPipeline.ask`

SSE-протокол (ADR-025), блоки (ADR-024), follow-ups не меняются — они работают на уровне `RagResult`.

### Интеграция в ChatRoutes

```kotlin
when (val d = router.decide(question)) {
    Irrelevant  -> respond(400, "Этот чатбот отвечает только...")
    GenericRag  -> stream(ragPipeline.ask(question))
    is UseSkill -> stream(skillExecutor.execute(skills[d.name]!!, question))
}
```

Остальной код стрима (text-блоки, sources, suggestions, `[DONE]`) идентичен для обоих путей.

## Обоснование

- **Точечные и обзорные вопросы требуют разных механизмов ответа.** Увеличивать top_K не решает проблему — модель тонет в нерелевантных чанках. Выделить обзорный путь явно — честнее.
- **Declarative > imperative.** Skill — это markdown-файл с двумя полями конфига и промптом. Новый класс вопросов = новый файл, не новый код. Расширяется без перекомпиляции.
- **Экономия вызовов.** Guard и router — одна и та же задача классификации (отсечь + разветвить). Слияние в один LLM-вызов экономит ~200ms и снижает стоимость.
- **Переиспользуем инфраструктуру.** `RagResult`, SSE-стриминг, UI-блоки, circuit breaker — всё остаётся. Skill — это альтернативный retrieval, не альтернативный транспорт.
- **Fail fast над fallback.** Если skill перегружает контекст — ошибка, а не тихое переключение. Скрытый fallback маскирует баг конфига и откладывает починку.

## Последствия

### Положительные
- Обзорные вопросы получают полные, структурированные ответы.
- Новые классы вопросов заводятся правкой одного файла — не кода.
- Guard и router — один вызов, не два.
- Skill-инструкции явно декларируют, как отвечать на данный класс — аудируемо и редактируемо.

### Отрицательные / Компромиссы
- Контекст skill'а может быть большим (до 100K токенов) — стоимость ответа выше обычного RAG (~3-5× за обзорные запросы). Это осознанно: пользователь, спросивший «все проекты», ожидает полного ответа, а не экономии.
- Классификатор может ошибиться: скилл активируется на не-обзорном вопросе, или наоборот не активируется когда надо. Смягчение — триггеры в skill-файле и `GENERIC` как безопасный дефолт.
- Skill-файлы — новая сущность, требующая поддержки: если переименовать директорию в `docs/`, пути в skills тоже нужно обновить. Валидация путей на старте (warn, если директория не найдена) снижает риск.
- QuestionGuard renamed → QuestionRouter. Breaking change для внутреннего API, но внешне (HTTP) ничего не меняется.

## Рассмотренные альтернативы

### Увеличить top_K до 20-30
Отклонено: (1) не решает кейсы, где нужен полный обход (5 проектов из 9 — всё равно неполный обзор); (2) ломает точечные вопросы — лишние чанки съедают контекст и размывают ответ; (3) стоимость растёт везде, не только для обзоров.

### Отдельный «overview» индекс с чанками-аннотациями
Идея: хранить по каждому файлу короткую аннотацию («Проект МРС, tech lead, low-code платформа»), на обзорных вопросах возвращать все аннотации. Отклонено: требует поддерживать отдельный слой данных, синхронизировать с основным, и всё равно не решает кейсы, где нужен полный текст. Skill-путь проще и делает то же самое без нового индекса.

### Tool calling — skill как tool, который LLM вызывает сама
LLM решает: «это обзорный вопрос — вызываю tool `list_projects`». Отклонено: (1) +1 LLM round-trip (сначала tool_call, потом ответ с контекстом); (2) LLM нестабильно решает, когда вызывать — нужен prompt engineering поверх prompt engineering; (3) у нас уже есть guard, дешевле расширить его. Может быть рассмотрено в будущем, если skill'ов станет >20 и классификатор начнёт путаться.

### Прицепить skill-instruction к обычному RAG, не меняя retrieval
Полу-мера: на обзорный вопрос добавляем инструкцию «структурируй», но retrieval всё равно возвращает top-5. Отклонено: инструкция без данных = пустые слова. Обзор невозможен без полного текста.

### Embedding-matching триггеров (без LLM-вызова)
Эмбеддить triggers на старте, сравнивать с вопросом по cosine. Отклонено для MVP: (1) требует калибровки порогов, (2) плохо на парафразах и опечатках, (3) у нас и так есть LLM-вызов (guard) — бесплатно расширить его до роутинга дешевле. Может быть добавлено позже как быстрый префильтр перед LLM-классификатором.

### Fallback на generic RAG при переполнении контекста skill'а
Отклонено: маскирует конфигурационную ошибку. Skill сломан → чинить skill. Тихий fallback означает, что пользователь получает неполный ответ, а разработчик не узнаёт о проблеме.

## Связь с другими ADR

- **ADR-016** (RAG pipeline) — skill не заменяет RAG, а добавляет альтернативный путь ответа для класса вопросов, которые retrieval по природе не покрывает.
- **ADR-019** (multi-stage RAG) — не затронут, обычный путь `GenericRag` работает по-прежнему через HyDE + BM25 + RRF.
- **ADR-024/025** (UI-блоки, единая модель ответа) — SkillExecutor возвращает тот же `RagResult`, механизмы извлечения блоков и стриминг не меняются.
- **ADR-017** (API protection) — circuit breaker из QuestionGuard переезжает в QuestionRouter как есть.
- **ADR-023** (LLM fallback) — механизм fallback моделей работает для skill-стрима так же, как для обычного RAG.
