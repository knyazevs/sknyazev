# ADR-11: CI/CD и деплой

- Статус: Принято
- Дата: 2026-04-01
- Авторы: Sergey Knyazev, Claude Sonnet 4.6
- Теги: ci-cd, деплой, github-actions, github-pages, инфраструктура

## Контекст

Проект состоит из двух частей с разным способом деплоя:

- **Фронтенд** (Astro + Svelte) — статическая сборка, хостинг GitHub Pages
- **Бэкенд** (AI-прокси: LLM, TTS, ASR) — сервис на VPS

Репозиторий публичный. Процесс деплоя должен быть прозрачным
и сам по себе является частью публичного кейса.

---

## Решение

### Фронтенд: GitHub Actions → GitHub Pages

Деплой фронтенда автоматизирован через **GitHub Actions workflow**.

Триггер: push в ветку `main`.

Шаги:
1. Checkout репозитория
2. Установка зависимостей
3. Сборка Astro (`astro build`)
4. Публикация артефакта `dist/` на GitHub Pages через `actions/deploy-pages`

Конфигурация хранится в `.github/workflows/deploy.yml`.

### Бэкенд: деплой на VPS вне scope GitHub Pages

Деплой бэкенда управляется отдельно (VPS).
Стратегия деплоя бэкенда фиксируется при необходимости отдельным ADR.

---

## Обоснование решения

GitHub Actions — нативный инструмент для репозитория на GitHub.
Не требует сторонних CI-сервисов. Workflow виден в публичном репозитории
и сам является частью демонстрации инженерной культуры.

GitHub Pages — бесплатный хостинг статики, достаточный для фронтенда.
Нет необходимости в платных платформах (Vercel, Netlify) для статического вывода.

---

## Альтернативы и причины отказа

### Vercel / Netlify
**Отказано**: дополнительная зависимость от внешней платформы без выигрыша
для статического сайта. GitHub Pages полностью покрывает задачу.

### Ручной деплой
**Отказано**: не соответствует стандарту production-grade проекта,
противоречит демонстрационным целям.

### Monorepo-деплой (фронт + бэк в одном workflow)
**Отказано на старте**: разные целевые среды (GitHub Pages vs VPS) требуют
разных механизмов. Совмещение усложняет без выигрыша.

---

## Последствия

### Положительные
- Автоматический деплой фронтенда при каждом merge в `main`
- Workflow публично виден — часть открытого кейса
- Нет зависимости от сторонних CI-платформ

### Отрицательные / Стоимость
- Бэкенд деплоится отдельно — нет единого пайплайна для всей системы
- При ошибке сборки GitHub Pages остаётся на предыдущей версии (нет explicit rollback)

---

## Примечание

Конфигурация workflow хранится в `.github/workflows/deploy.yml`
и является частью публичного источника истины проекта.
