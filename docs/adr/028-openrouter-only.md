# ADR-28: Единый провайдер — только OpenRouter, отказ от OpenAI напрямую

- Статус: Принято
- Дата: 2026-04-23
- Авторы: Sergey Knyazev, Claude Opus 4.7
- Теги: infra, llm, rag, embedding, provider

## Контекст

В `server/` одновременно жили два HTTP-клиента к LLM-провайдерам:

- `OpenRouterClient` — чат-стриминг, classifier, agentic tool-use (ADR-024, ADR-027). Использует `OPENROUTER_API_KEY`, ходит в `https://openrouter.ai/api/v1`.
- `OpenAiClient` — эмбеддинги (`text-embedding-3-small`), TTS (`tts-1`), ASR (`whisper-1`). Использует отдельный `OPENAI_API_KEY`, ходит в `https://api.openai.com/v1`.

Схема появилась органически: сначала был только OpenAI, потом ради fallback-цепочки моделей (ADR-023) и доступа к Anthropic добавился OpenRouter. Эмбеддинги и голосовые API на момент ADR-023 через OpenRouter были недоступны, поэтому `OpenAiClient` остался.

Два разных ключа и двух провайдеров неудобно:

- дублируется биллинг и квоты (два счёта, два лимита)
- в CI/CD два секрета (`OPENROUTER_API_KEY`, `OPENAI_API_KEY`), легче забыть ротацию
- конфиг рассыпан по двум секциям (`app.llm.*`, `app.embedding.*`) с разными url'ами
- кандидат на ещё один провайдер (Fireworks, Together) повторит эту же ось

OpenRouter сейчас отдаёт `/v1/embeddings` как OpenAI-совместимый прокси (модели вида `openai/text-embedding-3-small`, с обязательным префиксом провайдера). TTS/ASR через OpenRouter не проксируются.

## Решение

Единственный провайдер — OpenRouter. Один ключ `OPENROUTER_API_KEY` для всего: чат, classifier, эмбеддинги.

### Что меняется в коде

- `OpenRouterClient.embedText(text)` — новый метод, POST на `${embeddingBaseUrl ?: baseUrl}/embeddings`, формат совместим с OpenAI.
- `EmbeddingService` теперь принимает `OpenRouterClient` вместо `OpenAiClient`.
- `Bootstrap.kt` перестаёт инстанцировать `OpenAiClient`.
- `AppConfig.openAiApiKey` удалён. `app.embedding.baseUrl` по умолчанию = `https://openrouter.ai/api/v1`.
- В `docker-compose.prod.yml` и `deploy-server.yml` удалены переменные `OPENAI_API_KEY`.
- Модель эмбеддинга стала `openai/text-embedding-3-small` (формат OpenRouter с обязательным префиксом провайдера). От апгрейда до `-large` отказались: прирост качества retrieval'а маргинальный, а стоимость токена ×6.5. На нашем корпусе качество сильнее зависит от чанкинга и HyDE, чем от размерности вектора.

### Что временно ломается

TTS (`/api/tts`) и ASR (`/api/asr`) через OpenRouter недоступны — их регистрация в `Routing.kt` закомментирована. Код `OpenAiClient`, `TtsRoutes`, `AsrRoutes` оставлен в репозитории: если/когда голосовые API появятся через OpenRouter (или выберется другой провайдер), вернуть роуты — один коммит.

Фронтенд (`RightPanel.svelte`) продолжит слать запросы на `/api/tts` и `/api/asr` — они будут возвращать 404. Пока фронт не трогаем: пользователь всё ещё может писать текстом, а запись голоса не критична для демо-стенда.

### Инвалидация embedding-кеша

`EmbeddingCache` ключует вектора по (filePath, embeddingModel). Смена имени с `text-embedding-3-small` на `openai/text-embedding-3-small` инвалидирует cache-key один раз — векторное пространство то же самое (та же OpenAI-модель через прокси), но ключ кеша поменялся. Разовый cold-start ~20 секунд на весь корпус.

## Альтернативы и причины отказа

**Оставить два провайдера.** Status quo. Отказ: два ключа, два биллинга, два url'а в конфиге — лишний когнитивный налог ради API, которые сейчас отключены (голос) и стабильны (эмбеддинги).

**Удалить TTS/ASR код полностью.** Голос перестал работать две итерации назад и не возвращался. Отказ: оставляю явным TODO (закомментированная регистрация + ADR) — чтобы через полгода помнить, что фича существовала и почему отключена.

**Заменить эмбеддинги на локальную модель (ONNX/KInference).** Теоретически идеально: нулевая стоимость, полная независимость. Отказ: вес модели добавит ~100MB к образу, на K/N-стеке KInference-интеграция — отдельный проект, а корпус 90К токенов эмбеддится за 20 секунд один раз при cold-start — не узкое место.

**Свой прокси-сервис перед OpenRouter.** Для единой точки логирования/ретраев. Отказ: преждевременная оптимизация, текущего fallback-механизма (ADR-023) достаточно.

## Последствия

- Операционно проще: один секрет в secrets, один биллинг, одна точка отказа.
- RAG-пайплайн не меняется по качеству (та же модель эмбеддинга, просто через прокси).
- Голосовой UX временно не работает. Когда появится альтернатива — включить обратно через раскомментирование двух строк в `Routing.kt`.
- При добавлении нового провайдера (например, локального Ollama) — ось уже правильная: `OpenRouterClient` — единственный боевой, остальные проксируются через него или добавляются как feature-flag.
