# ADR-12: Стратегия чтения документации

- Статус: Принято
- Дата: 2026-04-01
- Авторы: Sergey Knyazev, Claude Sonnet 4.6
- Теги: архитектура, документация, github-api, build, dev

## Контекст

Левая панель должна отображать содержимое репозитория (ADR, диаграммы, README и др.)
без дублирования и хардкода структуры в коде.

Два режима работы предъявляют разные требования:

- **Dev**: скорость, работа offline, нет зависимости от сети
- **Prod**: актуальность в реальном времени — новый файл появился в репо,
  сайт показывает его без пересборки и редеплоя

---

## Решение

Использовать разные источники в зависимости от окружения,
за единым клиентским интерфейсом:

### Dev — Vite middleware plugin

Vite dev server дополняется кастомным middleware, который:
- читает файловую структуру `docs/` через Node `fs`
- отдаёт дерево документов и содержимое файлов через `GET /api/docs/*`
- не требует сетевых запросов и дополнительных серверов

### Prod — GitHub Contents API

Левая панель (Svelte island) напрямую обращается к публичному GitHub API:
- `api.github.com/repos/{owner}/{repo}/contents/{path}` — дерево файлов
- `raw.githubusercontent.com/{owner}/{repo}/{branch}/{path}` — содержимое файлов

Репозиторий публичный, auth не требуется.
Rate limit GitHub API: 60 req/ч без токена — достаточно для портфолио.

### Единый интерфейс

Клиентский модуль `docs-client.ts` абстрагирует оба источника:
- `import.meta.env.DEV === true` → запросы к `/api/docs/*` (Vite middleware)
- `import.meta.env.DEV === false` → запросы к GitHub API

Левая панель не знает об источнике данных.

---

## Поток данных

```text
Dev:
  LeftPanel (Svelte) → docs-client → GET /api/docs/* → Vite plugin → fs → docs/

Prod:
  LeftPanel (Svelte) → docs-client → GitHub API → raw.githubusercontent.com
```

---

## Обоснование решения

### Почему не статическая генерация (build-time)

Статическая генерация требует пересборки и редеплоя при каждом изменении документации.
Это противоречит цели: документация должна жить отдельно от деплоя фронтенда.

### Почему GitHub API напрямую, без VPS

Для публичного репо и трафика портфолио GitHub API не требует auth.
VPS как прокси добавляет операционную сложность без ощутимой выгоды.
При необходимости (превышение rate limit) VPS-прокси добавляется без изменения клиента.

### Почему Vite plugin, а не отдельный сервер в dev

Vite middleware встраивается в уже запущенный dev server.
Не нужно запускать два процесса. Прозрачно для разработчика.

---

## Ограничения

- В prod требует подключения к интернету (GitHub API)
- Rate limit: 60 req/ч без токена (при необходимости решается VPS-проксированием)
- Репозиторий должен быть публичным

---

## Последствия

### Положительные
- Документация обновляется в реальном времени без пересборки фронтенда
- Структура документов не хардкодится в коде
- Dev и prod используют один и тот же код левой панели

### Отрицательные / Стоимость
- Левая панель — Svelte island (требует JS), не zero-JS Astro компонент
- Первый рендер зависит от скорости GitHub API в prod

---

## Примечание

Данный подход отражает принцип ADR-2:
источник истины (репозиторий) остаётся единственным источником,
а UI является производным представлением — теперь в реальном времени.
