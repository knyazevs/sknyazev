# ADR-8: Инфраструктура AI, голоса и речи

- Статус: Принято
- Дата: 2026-04-01
- Авторы: Sergey Knyazev, Claude Sonnet 4.6
- Теги: llm, tts, asr, инфраструктура, голос, openrouter

## Контекст

Правая часть системы требует трёх AI-компонентов:

1. **LLM** — генерация ответов на вопросы об авторе
2. **TTS (Text-to-Speech)** — озвучивание ответов голосом
3. **ASR (Automatic Speech Recognition)** — распознавание голосового ввода пользователя

Рынок AI-сервисов нестабилен: модели меняются, цены меняются,
появляются новые провайдеры. Жёсткая привязка к одному провайдеру
создаёт риск вынужденной миграции с переработкой кода.

Голосовой интерфейс — центральный элемент первого впечатления.
Качество голоса напрямую влияет на восприятие проекта.
Компромисс по качеству ради экономии неприемлем.

---

## Решение

### LLM: OpenRouter как единая точка доступа

Все запросы к LLM направляются через **OpenRouter** — агрегатор,
предоставляющий унифицированный API для множества моделей
(OpenAI, Anthropic, Mistral, локальные модели и др.).

Это позволяет:
- переключать модель без изменения кода
- тестировать разные модели на одном контенте
- в перспективе — подключить собственный self-hosted inference

### TTS: OpenAI TTS по умолчанию, архитектура замены

По умолчанию используется **OpenAI TTS** (качество достаточно для wow-эффекта,
разумная стоимость, низкая latency через VPS-бэкенд).

TTS-компонент реализуется за абстракцией (интерфейс провайдера),
что позволяет переключиться на ElevenLabs, Azure Cognitive Speech
или self-hosted решение без изменения остального кода.

Web Speech API **не используется** — качество несовместимо с требованиями проекта.

### ASR: Whisper

Для распознавания речи используется **Whisper** (OpenAI).
Может быть запущен на VPS-бэкенде (self-hosted) или через API.
Высокое качество распознавания, поддержка русского языка.

### Инфраструктура: GitHub Pages + VPS-бэкенд

Все вызовы к AI-сервисам проходят через **собственный VPS-бэкенд**:
- API-ключи не экспонируются в клиентском коде
- Возможность rate limiting и логирования
- Единое место для смены провайдеров

Фронтенд деплоится на **GitHub Pages** (статика).
Бэкенд — отдельный сервис на VPS.

## Обоснование решения

### Почему OpenRouter, а не прямой OpenAI/Anthropic

Рынок LLM меняется каждые несколько месяцев.
OpenRouter позволяет менять модели без технического долга.
Это применение принципа "откладывай принятие решений" к выбору AI-провайдера.

### Почему не Web Speech API

Голосовой интерфейс — это первое впечатление посетителя.
Роботизированный голос браузерного TTS разрушает wow-эффект
и противоречит задаче проекта. Вопрос не в стоимости, а в качестве.

### Почему абстракция над TTS

ElevenLabs предлагает более высокое качество, но ограниченный бесплатный tier
и более высокую стоимость. Архитектура позволяет начать с OpenAI TTS
и переключиться при необходимости без переработки.

---

## Альтернативы и причины отказа

### Прямые вызовы OpenAI/Anthropic без OpenRouter
**Отказано**: жёсткая привязка к провайдеру, миграция потребует переработки.

### Web Speech API (браузерный TTS)
**Отказано**: неприемлемое качество голоса для целей проекта.

### Полностью self-hosted LLM
**Отказано на старте**: значительные затраты на GPU-инфраструктуру,
нецелесообразно для MVP. Остаётся опцией через OpenRouter.

---

## Последствия

### Положительные
- Гибкость смены AI-провайдеров без технического долга
- Качество голоса соответствует wow-эффекту
- API-ключи защищены бэкендом
- Путь к self-hosted inference открыт

### Отрицательные / Стоимость
- Зависимость от OpenRouter как промежуточного сервиса
- Бэкенд на VPS требует поддержки (uptime, обновления)
- Latency чуть выше из-за дополнительного хопа через VPS

---

## Примечание

Выбор OpenRouter отражает принцип: откладывай необратимые решения.
В быстро меняющемся пространстве AI-моделей это снижает технический долг
и сохраняет архитектурную гибкость.
