# ADR-5: UI и композиционная модель правой части (layout, состояния, потоки, деградация, доступность)

- Статус: Принято
- Дата: 2026-01-02
- Авторы: Sergey Knyazev, Claude Sonnet 4.6
- Теги: ui, композиция, состояние, доступность, взаимодействие, архитектура

## Контекст

Правая часть системы является интерактивным представлением: она отвечает на вопросы, озвучивает ответы, показывает субтитры и визуальную реакцию абстрактной сущности. Эта часть должна демонстрировать «исполнение» архитектурного замысла и при этом оставаться спокойной, предсказуемой и инженерно управляемой.

Особенности правой части:

- несколько параллельных каналов взаимодействия: текстовый ввод, голосовой ввод, голосовой вывод, субтитры, визуальная реакция
- риск расползания сложности UI в «кучу виджетов» без доминирующей композиции
- необходимость жёсткой трассируемости: UI — представление, а не источник истины
- необходимость деградации: система обязана работать без микрофона, без TTS, без звука
- необходимость прозрачных состояний: пользователь должен понимать «что сейчас происходит»

## Решение

Определить правую часть как композицию из трёх постоянных областей и одного слоя состояния:

1. **Центральная сцена (Scene):** абстрактная визуальная сущность (реактивная форма)
2. **Субтитры (Captions):** отображение текущего произносимого/показываемого текста
3. **Полоса ввода (Composer):** текстовое поле ввода + управляющие действия (например: отправить, остановить, включить/выключить микрофон)
4. **Слой состояния (State Layer):** единая модель состояний и событий, управляющая поведением всех областей

Композиция считается «стабильной»: зоны присутствуют всегда, меняется их содержимое и акценты в зависимости от состояния.

## Компоновка (Layout)

### Базовая структура (правая часть)

- **Центр:** визуальная сущность (максимальный визуальный приоритет)
- **Низ:** субтитры (читабельный, стабильный блок)
- **Самый низ:** поле ввода (всегда доступно; не исчезает, кроме случаев явного фокуса на доступность)

Цель: пользователь всегда видит «куда вводить» и «что система сказала», а визуальная сущность остаётся главным носителем динамики.

## Состояния UI

Правый интерфейс следует модели состояний, совместимой с ADR-4:

- `Idle` — ожидание
- `Listening` — активен микрофон (если доступен)
- `Thinking` — подготовка ответа
- `Speaking` — воспроизведение ответа (TTS) или демонстрация ответа (если без TTS)
- `Error` — ошибка (с деградацией)

### Отображение по состояниям (UI contract)

#### `Idle`
- Scene: спокойная анимация ожидания
- Captions: пусто или последняя завершённая реплика (по UX-выбору)
- Composer: доступен; можно печатать и отправлять

#### `Listening`
- Scene: выраженный, но не агрессивный индикатор «слушаю» (без антропоморфизма)
- Captions: показывает «Слушаю…» и/или черновик распознанного текста (если ASR включён)
- Composer: доступен (fallback), плюс кнопка «Остановить/Отменить»

#### `Thinking`
- Scene: индикатор активности (например, пульсация)
- Captions: краткое состояние «Формирую ответ…» (без лишней болтовни)
- Composer: доступен; кнопка «Отменить» (если поддерживается)

#### `Speaking`
- Scene: реактивная визуализация под аудио-вывод (или деградационный режим без аудио)
- Captions: вывод текущей фразы (базовый режим) или караоке (опционально)
- Composer: доступен; кнопка «Стоп» (остановить воспроизведение)

#### `Error`
- Scene: мягкая индикация ошибки (без «красных сирен»)
- Captions: понятное сообщение об ошибке + что можно сделать
- Composer: всегда доступен для продолжения текстом

## Потоки взаимодействия (User flows)

### Поток A: Текстовый ввод → текстовый/голосовой вывод
1. Пользователь вводит текст в Composer и отправляет
2. UI переходит в `Thinking`
3. После получения ответа:
   - если TTS доступен и включён → `Speaking` (голос + субтитры)
   - иначе → остаётся в «безголосом Speaking»: показ текста + анимация “говорит/молчит”
4. Завершение → `Idle`

### Поток B: Голосовой ввод → текстовый/голосовой вывод
1. Пользователь включает микрофон → `Listening`
2. Получение текста запроса → `Thinking`
3. Далее как в потоке A

### Поток C: Прерывание
- В `Listening`: кнопка «Стоп» прекращает запись/распознавание и возвращает в `Idle`
- В `Speaking`: кнопка «Стоп» прекращает воспроизведение и возвращает в `Idle` (или оставляет текст ответа в Captions)

## Правила деградации (Degradation rules)

Деградация считается частью архитектуры UI, а не “планом Б”.

1. **Нет микрофона/разрешения**
   - голосовой ввод недоступен
   - Composer остаётся основным входом
   - UI не должен давить на пользователя запросами разрешений без действия пользователя

2. **Нет TTS или пользователь отключил звук**
   - канал вывода: текст (Captions) становится основным
   - Scene работает в режиме без анализа аудио: анимация по событиям “говорит/молчит” или по текстовой эвристике
   - никаких “пустых ответов”: текст обязателен

3. **Плохая сеть/ошибка распознавания/ошибка TTS**
   - система обязана завершать запрос с понятным состоянием и вернуть контроль пользователю
   - диалог продолжается через текст

## Доступность (A11y)

Минимальные обязательства:

- всё функционально без звука и без микрофона
- субтитры читабельны (контраст, размер, переносы строк)
- управление доступно с клавиатуры
- чёткие состояния: пользователь понимает, когда активен микрофон и когда идёт воспроизведение
- отключаемые анимации (уменьшение движения), если платформа/настройки пользователя это требуют

## Разделение ответственности (UI boundaries)

UI правой части:
- отображает состояния и результаты
- запускает/останавливает ввод и вывод (как контроллер представления)
- не является местом, где рождаются факты или изменяется источник истины

Контент (текст вопроса/ответа) считается каноническим в рамках сессии; голос и визуальная реакция — производные представления (см. ADR-4).

## Последствия

### Положительные
- Появляется стабильная композиция, которую легко объяснить и развивать.
- Состояния и потоки формализованы, что снижает хаос и гонки UI.
- Доступность и деградация встроены в контракт интерфейса.
- Визуальная сущность сохраняет центральную роль без превращения в «аватара».

### Отрицательные / компромиссы
- Требуется аккуратная реализация state-machine и отменяемых операций.
- Караоке-режим остаётся опциональным и может быть недоступен без таймкодов.
- Нужно дисциплинированно поддерживать согласованность между Captions/TTS/Scene.

## Рассмотренные альтернативы

### Альтернатива A: Чат-ориентированный UI (список сообщений как главный объект)
**Отклонено**, потому что:
- смещает восприятие в «обычный чат-бот»
- визуальная сущность становится вторичной декорацией
- теряется идея «голоса системы» как центрального опыта

### Альтернатива B: Полный экран для визуализации, а ввод/текст по модальным слоям
**Отклонено**, потому что:
- ухудшается доступность и предсказуемость (пользователь “теряет” поле ввода)
- усложняется навигация и контроль состояния

### Альтернатива C: Скрывать субтитры, если включён звук
**Отклонено**, потому что:
- субтитры — не “fallback”, а полноценный канал доверия и проверяемости
- без субтитров сложнее контролировать точность и понимание ответа
