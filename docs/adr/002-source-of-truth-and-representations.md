# ADR-2: Источник истины и модель представлений

- Статус: Принято
- Дата: 2026-01-02
- Авторы: Sergey Knyazev, LLM (ChatGPT 5.2), Claude Sonnet 4.6
- Теги: архитектура, данные, представление, ai

## Контекст

Проект одновременно выполняет несколько ролей:
- персональный сайт
- архитектурная документация
- демонстрационный инженерный проект
- интерактивный интерфейс с LLM и голосовым взаимодействием

Эти роли требуют **разных форм представления информации**:
- строгая, проверяемая документация
- интерактивное и диалоговое объяснение
- визуальные и аудиальные формы

Без явного разделения между:
- тем, *что является истиной*
- и тем, *как это показывается и интерпретируется*

возникают риски:
- LLM начинает «додумывать»
- UI подменяет факты интерпретациями
- документация деградирует в маркетинг
- проект теряет доверие как инженерный артефакт

---

## Решение

Ввести жёсткое архитектурное разделение между:

1. **Источником истины (Source of Truth)**
2. **Слоями представлений (Representations)**

Источник истины является:
- неизменяемым через UI и LLM
- единственным местом, где фиксируется опыт
- основой для всех остальных слоёв

Все представления:
- являются производными
- могут интерпретировать, но не изменять данные
- обязаны быть трассируемыми к источнику истины

---

## Источник истины (Source of Truth)

### Состав

Источник истины включает:
- ADR-документы
- архитектурные описания
- диаграммы (UML, C4, sequence и т.п.)
- зафиксированные факты опыта
- структурированные доменные данные

### Формат

- текстовые файлы (Markdown)
- хранятся в репозитории
- версионируются
- доступны для прямого просмотра

### Свойства

Источник истины:
- детерминирован
- проверяем
- воспроизводим
- независим от UI и LLM

---

## Слои представлений (Representations)

### 1. Документальное представление (левая часть интерфейса)

Назначение:
- показать внутреннюю структуру проекта
- продемонстрировать инженерную дисциплину
- дать доступ к первоисточникам

Содержит:
- ADR
- диаграммы
- архитектурные пояснения
- связи между решениями

Характер:
- статичное
- точное
- без интерпретаций

---

### 2. Интерактивное представление (правая часть интерфейса)

Назначение:
- объяснять
- отвечать на вопросы
- показывать результат архитектурных решений
- обеспечивать голосовое и текстовое взаимодействие

Форма:
- LLM-агент
- визуальная абстрактная сущность
- голосовой ввод и вывод
- субтитры / текст ответа

Характер:
- динамичное
- адаптивное
- интерпретирующее

---

## Роль LLM в системе

LLM:
- **не является источником истины**
- не имеет права добавлять новые факты
- не может изменять доменные данные
- работает только на основе предоставленного контекста

Функции LLM:
- навигация по источнику истины
- объяснение архитектурных решений
- адаптация уровня детализации под пользователя
- перевод строгой документации в диалоговую форму

LLM всегда действует:
> «на основании задокументированных решений автора»

---

## Поток данных

```text
Source of Truth (ADR, diagrams, data)
        ↓
 Structured Context
        ↓
 LLM Interpretation Layer
        ↓
 UI / Voice / Visual Representation
```

Обратного потока **не существует**.

---

## Ограничения

* UI не может редактировать источник истины
* LLM не может сохранять новые знания
* любые изменения опыта фиксируются только через:

  * новые ADR
  * обновления документации

---

## Последствия (Consequences)

### Положительные

* Высокий уровень доверия к контенту
* Контролируемое использование AI
* Чёткое разделение ответственности
* Масштабируемость проекта

### Отрицательные

* Увеличение объёма документации
* Более сложная архитектура
* Требуется дисциплина поддержки

Эти издержки осознаны и приняты.

---

## Примечание

Данное разделение подчёркивает, что:

* опыт принадлежит человеку
* интерпретация — инструмент
* технология служит объяснению, а не подмене смысла
