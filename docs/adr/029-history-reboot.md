# ADR-29: Перезагрузка истории коммитов и переезд на `knyazevs.github.io`

- Статус: Принято
- Дата: 2026-04-23
- Авторы: Sergey Knyazev, Claude Opus 4.7
- Теги: process, repo, meta

## Контекст

Репозиторий прошёл фазу проверки гипотезы: цель была быстро получить работающий прототип — орб, чат, RAG, сквозной пайплайн от вопроса до ответа со ссылками. На этой фазе я сознательно не тратил ресурс на гигиену истории: скорость изменений была важнее читаемости лога.

Как следствие, история из 41 коммита содержит сообщения вида `big update`, `blocks`, `Update Bootstrap.kt`, `wip`, `update ci/cd` × несколько. При этом в `CLAUDE.md` самого проекта прописаны жёсткие правила коммитов (`<тип>(<область>): <что сделано>`, обязательные ADR, обязательный блог-пост на каждое арх-решение) — и текущая история прямо противоречит заявленным правилам.

Проект позиционируется как публично верифицируемый артефакт для техлид/архитектор-роли. Сам `git log` — часть артефакта: рекрутёр/EM читает его как сигнал о рабочем процессе. Разрыв между декларацией (`CLAUDE.md`) и реальностью (log) обесценивает и то, и другое.

К этому моменту проект достиг зрелости, при которой дальнейшие изменения уже не должны быть «скоростными» — архитектура устоялась (ADR-001…028 покрывают основные решения), фичевый контур закрыт, дальше — дисциплинированное ведение и развитие.

## Решение

Выполнить **squash-reboot** истории с одновременным переездом в новый репозиторий `knyazevs/knyazevs.github.io`:

1. Все текущие незакоммиченные изменения (ADR-027, ADR-028, агентский контур, OpenRouter-консолидация и др.) коммитятся одним финальным прототипным коммитом в текущий репо `knyazevs/sknyazev`. Это фиксирует полный финал прототипной фазы в архиве.
2. Старый репозиторий `knyazevs/sknyazev` остаётся как есть — полная история прототипа сохраняется в нём по прежнему URL и служит архивом.
3. Local `origin` переключается на `git@github.com:knyazevs/knyazevs.github.io.git`.
4. Локально создаётся orphan-ветка с текущим деревом → один baseline-коммит → force-push в `main` нового репо.
5. Новый репо становится основным. С этой точки проект ведётся строго по `CLAUDE.md`: осмысленные коммиты в формате `<тип>(<область>): <что>`, ADR на каждое арх-решение, блог-пост на каждое решение.

Baseline-коммит ссылается на этот ADR и на блог-пост `2026-04-23-history-reboot.md` — факт перехода виден прямо в первом коммите новой истории, а не только в документации.

## Альтернативы и причины отказа

**Оставить историю как есть + писать блог «почему log выглядит так».** Самый честный вариант — ничего не ломать, объяснить текстом. Отказ: для техлид-роли одного текста недостаточно, рекрутёр всё равно увидит `big update` в логе и составит впечатление раньше, чем прочитает блог. Рефлексия сильнее, когда подкреплена действием.

**Interactive rebase с переписыванием сообщений всех 41 коммита.** Заменить каждое `fix stuff` на осмысленное сообщение задним числом. Отказ: во-первых, это фальсификация — я не помню настоящих причин трёх-четырёх старых коммитов, сообщения получились бы реконструкцией. Во-вторых, force-push переписанной истории выглядит «слишком ровно» — опытный интервьюер считывает это как подчистку. В-третьих, дата коммитов сохраняется, а сообщения — нет, это создаёт внутреннее противоречие.

**Полный re-init (`rm -rf .git && git init`) в том же репо.** Психологически чище, но: (а) GitHub contribution graph обнуляется, пропадает сигнал «работал над этим регулярно»; (б) даты уже написанных блог-постов (`2026-04-XX-*.md`) повисают в воздухе без подтверждения в git; (в) старый репо уже публичен и кто-то мог его видеть — расхождение палится. Отказ.

**Squash-reboot в том же репо (тег `v0-prototype` + orphan-baseline).** Рассматривал как основной вариант. Отказ в пользу переезда на новый репозиторий: разделение «архив прототипа / проект» по границе целых репо чище, чем по тегу внутри одного репо. Кроме того, имя `knyazevs.github.io` — канонический user-site pattern GitHub Pages, проекту он подходит лучше, чем `sknyazev`.

## Последствия

- **Для внешнего наблюдателя:** новый репозиторий стартует с одного baseline-коммита, дальнейшая история ведётся по правилам `CLAUDE.md`. Старый репозиторий остаётся доступен как архив прототипной фазы — факт перехода прозрачен, ничего не прячется.
- **GitHub contribution graph:** активность в старом репо сохраняется. В новом — стартует с baseline. Это ожидаемо и не ухудшает сигнал «работал регулярно», если смотреть профиль целиком.
- **Для инфраструктуры:** GitHub Pages, CNAME/кастомный домен, deploy-workflow (`deploy-server.yml`) и любые хардкоды URL, завязанные на имя репо `sknyazev`, нужно обновить отдельным проходом после переезда. Это разовая работа, не архитектурная.
- **Для процесса:** дисциплина `CLAUDE.md` перестаёт быть декларацией — с baseline она становится фактом. Любой новый коммит, не соответствующий формату, теперь аномалия в чистой истории, а не шум в и-так-грязной.
- **Необратимость:** операция с force-push в новый репо необратима в рамках самого репо, но архив прототипа в `knyazevs/sknyazev` страхует от потери данных.
