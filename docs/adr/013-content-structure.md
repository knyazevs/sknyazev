# ADR-13: Структура контентных разделов и формат для RAG

- Статус: Принято
- Дата: 2026-04-01
- Авторы: Sergey Knyazev, Claude Sonnet 4.6
- Теги: контент, структура, rag, домен, документация

## Контекст

Сайт должен содержать не только архитектурные решения (ADR),
но и профессиональный профиль автора: опыт, проекты, компетенции.

Требования:
- Контент хранится в репозитории как источник истины (ADR-2)
- Файлы читаются в реальном времени без пересборки (ADR-12)
- Тот же контент используется как контекст для LLM/RAG (ADR-3)
- Опыт защищён NDA — компании и клиенты не называются (ADR-1)

---

## Решение

Добавить четыре новых раздела в `docs/`:

```
docs/
├── profile/
│   └── index.md
├── experience/
│   └── NNN-<system-slug>.md
├── projects/
│   └── <slug>.md
└── skills/
    └── <domain>.md
```

Каждый файл содержит YAML front matter с метаданными для поиска
и Markdown-тело с содержательным описанием.

---

## Разделы

### profile/index.md — Профессиональный профиль

Единственный файл. Описывает автора как профессионала:
фокус, стиль работы, тип задач, что ищет в следующей роли.

```yaml
---
type: profile
---
```

Личные данные (не профессиональные) — вне домена (ADR-1).

---

### experience/NNN-slug.md — Опыт работы

**Один файл = одна значимая инициатива/система.**

Группировка по инициативе, а не по работодателю:
- лучше для RAG (поиск по типу задачи, а не по компании)
- решает NDA (нет привязки к названию компании)
- соответствует доменной модели (ADR-1): System — центральная сущность

```yaml
---
type: experience
system-type: <тип системы, например payment-platform>
context: startup | scaleup | enterprise
lifecycle-stage: greenfield | scaling | legacy-migration | sunset
author-role: decision-maker | influencer | builder | mixed
period: "ГГГГ–ГГГГ"
team-size: "<число> engineers"
tags: [домен, технология, паттерн]
---
```

Тело файла: контекст, роль автора, ключевые решения, результаты.
Технологии — в теле, не в front matter.

---

### projects/slug.md — Проекты

Pet projects и открытые работы. Включая этот сайт.

```yaml
---
type: project
status: live-case | live | completed | archived
visibility: public | private
tags: [технология, паттерн]
---
```

Статусы:
- `live-case` — проект демонстрируется прямо сейчас и сам является кейсом
- `live` — активно развивается, публичный
- `completed` — завершён
- `archived` — заброшен

---

### skills/domain.md — Компетенции

Один файл на домен компетенций. Без уровней (`expert/advanced`) —
уровень выводится из контекста применения в теле файла,
а не декларируется как самооценка.

```yaml
---
type: skill-domain
domain: <название домена>
tags: [технология, паттерн, подход]
---
```

Тело файла: что умею, где применял, примеры решений.
Ссылки на связанные experience/projects/ADR.

---

## Обоснование

### Почему группировка по инициативе, а не по работодателю

Для RAG важна тематическая близость: вопрос "опыт с legacy migration"
должен находить релевантный файл, а не список работодателей.
Группировка по инициативе даёт это нативно.

### Почему нет уровней в skills

Декларативный уровень (`expert`) — самооценка без доказательств.
Контекст применения в теле файла позволяет читателю
(и LLM) самому сделать вывод об уровне.

### Почему front matter вместо произвольного Markdown

Front matter даёт структурированные метаданные для:
- фильтрации при RAG-поиске (тип, теги, роль)
- построения навигации без парсинга тела файла
- расширения схемы без изменения существующих файлов

---

## Последствия

### Положительные
- Контент масштабируется: новый файл = новая единица знания для LLM
- Навигация строится автоматически из файловой структуры (ADR-12)
- Единый формат упрощает подготовку RAG-контекста
- NDA-ограничения соблюдены структурно

### Отрицательные / Стоимость
- Требует дисциплины: соблюдать front matter схему при добавлении файлов
- Технологии только в теле — нет возможности фильтровать по стеку через метаданные

---

## Примечание

Схема front matter намеренно минимальна.
Расширение (например, добавление поля `tech-stack`) —
через новый ADR при появлении конкретной потребности.
