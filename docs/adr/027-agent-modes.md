# ADR-27: Режимы работы агента — agentic, full-context, rag

- Статус: Принято
- Дата: 2026-04-23
- Авторы: Sergey Knyazev, Claude Opus 4.7
- Теги: rag, agent, tool-use, architecture, cost

## Контекст

RAG-стек в проекте разросся: HyDE (ADR-019) → dense + BM25 параллельно → RRF-слияние → cosine-фильтр → top-5 → сборка промпта. Каждый слой добавлялся осмысленно и решает конкретную проблему. Skills (ADR-026) добавили отдельный путь для обзорных запросов.

Но есть факт, который стоит проговорить: **весь корпус проекта — 67К слов ≈ 90К токенов** (docs/ + server/src/ + app/src/ + scripts/). Claude Opus 4.7 держит 200K. Корпус влезает целиком с двукратным запасом.

Когда retrieval строился (ADR-016), это был правильный выбор: корпус и модели были другие. Сейчас HyDE+BM25+RRF делает дорогую выборку там, где можно либо отдать модели весь корпус, либо позволить ей самой навигировать по нему. Это не значит, что RAG плох — это значит, что он больше не безусловный дефолт.

Две альтернативы, ставшие реальными:

1. **Full-context** — загрузить весь корпус в system prompt, отвечать одним LLM-вызовом. Prompt caching у Anthropic через OpenRouter делает это дешёвым на повторных запросах в пределах 5 минут.
2. **Agentic tool-use** — дать LLM инструменты `list_files`, `read_file`, `grep` и позволить самой решать, что читать. Классический паттерн «агент с файловыми тулами», которым работают Claude Code, Cursor, Devin.

Выбирать один из трёх и хоронить остальные — плохо. Full-context даёт лучшее качество на сложных вопросах, но дорогой. Agentic дешевле и прозрачнее (видно, что модель читала), но добавляет латентность tool-roundtrip'ов. RAG быстрый cold-start, но хуже на cross-document reasoning. Хочу все три с возможностью переключения.

## Решение

Три режима как взаимозаменяемые реализации одного контракта — `suspend fun ask(question: String): RagResult`.

### Режимы

```kotlin
enum class AgentMode { AGENTIC, FULL_CONTEXT, RAG }
```

| Режим | Дефолт для | Стоимость/запрос | Латентность |
|-------|-----------|------------------|-------------|
| `agentic` | публичный прод | ~$0.05 (Haiku), ~$0.15 (Opus) | 2-4 LLM-вызова |
| `full-context` | ручное переключение (premium) | ~$0.14 (cached Opus), ~$1.35 (cold) | 1 LLM-вызов |
| `rag` | legacy, сравнение качества | ~$0.08 (с HyDE) | 3-4 параллельных (HyDE + retrieval + gen) |

### Роутинг

`QuestionRouter` (ADR-026) остаётся как первый гейт. Трёхрежимность применяется **только к пути `GenericRag`** — skill-путь и Irrelevant не меняются:

```kotlin
when (routingDecision) {
    Irrelevant    -> 400
    UseSkill(n)   -> skillExecutor.execute(skills[n], question)  // без изменений
    GenericRag    -> when (mode) {
        AGENTIC      -> agenticPipeline.ask(question)
        FULL_CONTEXT -> fullContextPipeline.ask(question)
        RAG          -> ragPipeline.ask(question)  // legacy-путь
    }
}
```

Это сохраняет инвариант: все три пайплайна возвращают одинаковый `RagResult`, весь SSE-транспорт, UI-блоки, suggestions, follow-ups работают без изменений.

### Agentic — устройство

Системный промпт содержит **карту корпуса** (`docs/.corpus-map.md`, ~3-5К токенов), сгенерированную отдельным скриптом. Карта — это не выжимка, а индекс: список всех файлов с frontmatter-полями и одним предложением. Карта отдаётся LLM как stable-cacheable префикс.

Инструменты навигации:

| Tool | Аргументы | Назначение |
|------|-----------|------------|
| `list_files` | `path` | дерево поддиректории в sandbox'е |
| `read_file` | `path` | полное содержимое файла |
| `grep` | `pattern`, `path?` | литеральный поиск, возвращает строки с контекстом |
| `read_git_log` | `path`, `limit?` | коммит-история файла или директории |

**Sandbox**: разрешены пути в `docs/`, `server/src/`, `app/src/`, `scripts/`, `server/skills/`. Любой другой путь — ошибка, возвращаемая LLM как tool result. Никаких `..`, никаких абсолютных путей вне whitelist'а.

Цикл: `streamChat` с навигационными tools → LLM либо отдаёт текст (стримим клиенту), либо tool_call → исполняем → результат добавляется как `tool`-message → снова `streamChat`. Максимум 8 итераций (фактический hard cap на бесконечный цикл — на практике 2-4).

UI-тулы (`render_image`, `render_link_list` и т.д.) передаются в *финальный* вызов — тот, что пишет текстовый ответ пользователю. На промежуточных итерациях (когда LLM просто ищет инфу) UI-тулы не передаются — это экономит токены и снимает соблазн «отрендерить раньше времени».

### Full-context — устройство

Весь корпус в system prompt'е, помеченный как cacheable. Один `streamChat` с UI-тулами. Prompt caching у OpenRouter/Anthropic работает на префиксе сообщений: пока содержимое не меняется, следующий запрос в пределах 5 минут оплачивается по 10% ставке.

Состав контекста:
- system prompt (правила ответа, цитирование, формат — тот же, что в RAG)
- весь `docs/` (markdown-файлы, конкатенированные с breadcrumb'ами)
- ключевые файлы из `server/src/` и `app/src/` (пути из corpus-map)
- текущая дата

Не-cacheable часть — только пользовательский вопрос в конце. Это максимизирует hit rate.

### Режим выбирается клиентом

Клиент (браузер) передаёт `mode` в теле запроса `POST /api/chat`:

```json
{ "question": "...", "mode": "agentic" }
```

Если поле отсутствует — дефолт `AGENTIC`. Валидные значения — `agentic`, `full_context`, `rag`. Неизвестное — 400.

UI-селектор показывается **только в debug-режиме** (повторяет паттерн DebugLogStream). Пользователь в обычной сессии получает agentic без переключателя.

### Карта корпуса — генерируется, не курируется

`scripts/generate-corpus-map.ts` — Node-скрипт (как `build-cv.ts`), читает:
- `docs/**/*.md` — из frontmatter берёт `title`/`name`/`date`, плюс первую H1 и первое предложение после
- `server/src/**/*.kt`, `app/src/**/*.{svelte,ts,astro}` — берёт путь и KDoc/JSDoc верхнеуровневого класса/функции
- `git log --oneline` — последние 20 коммитов

Пишет в `docs/.corpus-map.md`. Файл не коммитится (`.gitignore`) — регенерируется на `bun run build` и локально при разработке. Сервер читает его на старте; если не найден — стартует с warning и использует пустую карту (agentic всё ещё работает, просто первый tool_call будет `list_files`).

Отдельная команда `bun run build:corpus-map`. Добавляется в composite `build`.

## Обоснование

- **Каждый режим хорош в своей зоне.** Agentic — дефолт по цене/прозрачности. Full-context — для сложных кросс-документных вопросов, где важна полнота. RAG — оставляем для сравнения и как свидетельство того, что стек эволюционировал.
- **Переключение без удаления — сигнал зрелости.** Для CV-проекта история «сначала построил RAG, потом померил и переключился на agentic» ценнее, чем «сделал что-то одно». RAG остаётся рабочим, не закомментирован, не удалён.
- **Карта, а не выжимка.** LLM-компрессия теряет факты и вызывает галлюцинации. Карта сохраняет все первичные факты (пути, даты, заголовки) без потерь. Агент читает первичный документ, а не чью-то интерпретацию.
- **Sandbox на уровне тула.** Безопасность агентских файловых тулов — не вторичная задача; whitelist директорий и отказ от `..` делаются в коде тула, не в промпте.
- **Mode только для Generic пути.** Skills (ADR-026) уже решают класс обзорных запросов явным маршрутом. Смешивать их с mode было бы дублированием логики.
- **Prompt caching как honest win.** 10× разница в цене full-context'а cold vs warm — реальная, не теоретическая. Это превращает «дорого» в «дорого только первый раз за 5 минут».

## Последствия

### Положительные
- Дефолтный режим дешевле текущего RAG (~$0.05 vs ~$0.08 per query) — отсутствие HyDE и параллельных retrieval'ов экономит больше, чем добавляют tool-roundtrip'ы.
- Прозрачность: в debug-режиме видно, какие файлы агент читал. Это и UX-плюс, и отладочный инструмент.
- Full-context как escape hatch для вопросов, где модель должна увидеть всё сразу.
- Новый файл в `docs/` → автоматически попадает в карту и в agentic retrieval. Ре-индексация не нужна — никаких эмбеддингов для agentic-режима.
- RAG, Skills, UI-блоки, SSE, follow-ups — всё остаётся работать как было.

### Отрицательные / Компромиссы
- Agentic добавляет латентность: 2-4 LLM-вызова вместо одного. На «расскажи про Kotlin-опыт» — 4-6 секунд вместо 2-3.
- Full-context стоит $1.35 на холодном старте. После 5 минут бездействия следующий запрос платит полную цену снова. Это делает режим неподходящим дефолтом для редких пользователей сайта.
- Sandbox в тулах требует поддержки: если добавится новая корневая директория (`data/`, например) — whitelist надо обновлять. Валидируется тестами на допустимые и недопустимые пути.
- Карта корпуса может устареть, если забыть перегенерить. Смягчение: включить в `bun run build`; дополнительно — добавить `last_generated` в шапку карты, чтобы при чтении на старте сервер мог warn'уть о старой карте.
- LLM может «заблудиться» в tool-loop (8 итераций подряд `list_files` без текста). Hard cap + warn в логах; на практике Opus/Sonnet таким не страдают, но поведение надо мониторить.
- Три пайплайна вместо одного — больше кода. Компенсация: общий интерфейс `AgentPipeline` и общий transport-слой в `ChatRoutes`.

## Рассмотренные альтернативы

### Оставить RAG как единственный путь, оптимизировать дальше
Отклонено: корпус уже целиком влезает в контекст. Оптимизировать retrieval над 90К-корпусом — техдолг в сторону сложности. Для Tech Lead'а решение «убрать слой, а не добавить» сильнее, чем «шестая стадия retrieval'а».

### Полностью заменить RAG на agentic, удалить старый код
Отклонено: (1) для CV-проекта удалить работающее архитектурное решение = удалить демонстрацию навыка его построить. (2) Реальная ценность легко теряется, если agentic окажется хуже на каком-то классе вопросов — хочется иметь быстрый откат. (3) RAG остаётся как baseline для сравнения.

### Только full-context, без agentic
Отклонено: $1.35 на холодном запросе. Для сайта, где частота запросов низкая (посетители-рекрутеры, а не активные пользователи), холодные запросы будут доминировать. Full-context как дефолт — дорого.

### Agentic без карты корпуса («pure exploration»)
Отклонено: первый `list_files` на каждом запросе — это гарантированный wasted roundtrip. Карта в system prompt'е предотвращает это. Генерация карты однократная и дешёвая.

### Сжатая LLM-выжимка вместо карты
Отклонено: выжимка теряет факты. На вопрос «с каким трафиком работал» LLM додумает из сжатого представления — confabulation. Карта сохраняет первичные данные без компрессии, детали добираются через `read_file`. См. диалог с автором 2026-04-23.

### Mode toggle на сервере (через env), а не на клиенте
Отклонено: режим — это клиентский выбор. Один и тот же сервер должен уметь обслуживать разных пользователей с разными режимами (например, я — в debug, посетитель — в дефолте). Env-переменная хороша только для `DEFAULT_MODE`.

### RAG + agentic гибрид (сначала retrieve, потом дать агенту найденное)
Отклонено: усложнение без явной выгоды. Agentic с картой уже имеет обзор корпуса; RAG добавил бы нерелевантные выборки. Если в какой-то момент agentic начнёт проигрывать RAG — вернёмся к идее гибрида.

### Fine-tuning / RAG-finetune под корпус
Отклонено: overkill для 90К-корпуса. Стоимость тренировки и сложность обновлений не оправдываются.

## Связь с другими ADR

- **ADR-016, ADR-019** (RAG pipeline, multi-stage RAG) — RAG остаётся как один из трёх режимов, не удаляется. Путь `mode=rag` использует ровно существующий `RagPipeline`.
- **ADR-026** (Skills) — маршрутизация через `QuestionRouter` не меняется. Skills работают независимо от mode.
- **ADR-024, ADR-025** (UI-блоки, единая модель ответа) — все три пайплайна возвращают `RagResult`, транспорт и UI-блоки работают одинаково.
- **ADR-023** (LLM fallback) — механизм fallback'ов моделей применяется в каждом режиме: `streamChat` не знает о режиме выше.
- **ADR-017** (API protection) — circuit breaker и rate limiting не меняются; mode не даёт обхода защит.
- **ADR-002** (source of truth) — карта корпуса генерируется из первичных файлов без промежуточных слоёв данных; первичные `docs/` остаются единственным источником истины.
