# ADR-9: Frontend-фреймворк и модель рендеринга

- Статус: Принято
- Дата: 2026-04-01
- Авторы: Sergey Knyazev, Claude Sonnet 4.6
- Теги: frontend, astro, svelte, архитектура, рендеринг

## Контекст

Сайт состоит из двух концептуально разных зон (ADR-7):

**Левая часть** — документация:
- преимущественно статичный контент (Markdown, ADR, диаграммы)
- минимальная интерактивность
- важна читаемость и скорость загрузки

**Правая часть** — интерактивный интерфейс:
- real-time анимации, реакция на аудио
- голосовой ввод/вывод, subтитры
- state machine (Idle / Listening / Thinking / Speaking / Error)
- высокая реактивность и сложность состояния

Ограничения хостинга: GitHub Pages требует статического вывода.

Дополнительный контекст: сайт является публичным кейсом.
Технологический выбор сам по себе является демонстрацией архитектурного мышления.

---

## Решение

Принять **Astro + Svelte islands** как основной фронтенд-стек.

- **Astro** управляет статической частью, роутингом и сборкой
- **Svelte** используется как island-фреймворк для правой интерактивной части
- Левая часть рендерится без JavaScript (zero hydration)
- Правая часть гидрируется полностью как один Svelte island

### Модель рендеринга

```text
Astro (static shell)
 ├── Левая колонка (Astro components, zero-JS)
 │    ├── Markdown-рендеринг ADR
 │    ├── Навигация по документации
 │    └── Контакты
 └── Правая колонка (Svelte island, client:load)
      ├── State machine (голосовые состояния)
      ├── Абстрактная визуальная сущность
      ├── Субтитры
      └── Composer (ввод текста/голоса)
```

Сборка производит полностью статический вывод, совместимый с GitHub Pages.

## Обоснование решения

### Почему Astro

Astro спроектирован именно для сайтов с большим объёмом контента и
точечной интерактивностью. Его islands architecture архитектурно совпадает
с концепцией проекта: статичный источник истины + интерактивное представление.

Левая часть не нуждается в JavaScript — Astro обеспечивает это нативно.
Это ускоряет загрузку и снижает сложность.

### Почему Svelte (а не React/Vue)

- Svelte компилируется в чистый JS без runtime — меньше bundle, выше производительность
- Анимации и реактивность — первоклассные примитивы языка
- Меньший когнитивный overhead для state management в сложных UI
- Выбор Svelte вместо React — намеренный архитектурный сигнал:
  "автор выбирает инструмент под задачу, а не под популярность"

### Почему islands, а не SPA

SPA (например Next.js) загрузил бы весь фреймворк ради статического контента
левой части. Islands позволяют платить за JavaScript только там, где он нужен.

## Альтернативы и причины отказа

### Next.js / Nuxt
**Отказано**: избыточен для статического контента левой части,
SSR не нужен при хостинге на GitHub Pages,
React не даёт архитектурного сигнала о осознанном выборе.

### SvelteKit (без Astro)
**Отказано**: SvelteKit хорошо справился бы с задачей,
но не даёт нативного zero-JS для левой части.
Astro + Svelte архитектурно точнее отражает разную природу двух зон.

### Vue / Nuxt
**Отказано**: нет преимуществ перед Svelte для данной задачи,
менее выразителен для сложных анимаций.

### Vanilla JS + Web Components
**Отказано**: высокий объём boilerplate для state machine
правой части, нет экосистемы для анимаций.

## Последствия

### Положительные
- Архитектура рендеринга отражает концепцию проекта
- Левая часть — быстрая, без JS, SEO-friendly
- Правая часть — изолирована и полностью реактивна
- Статический вывод совместим с GitHub Pages
- Технологический стек является самостоятельным архитектурным сигналом

### Отрицательные / Стоимость
- Astro + Svelte — менее распространённая комбинация, меньше готовых примеров
- Потребуется настройка интеграции Svelte-island в Astro-проекте
- Мобильная адаптация сплит-лэйаута требует отдельного решения

---

## Примечание

Выбор Astro + Svelte islands — это не следование тренду.
Это применение принципа "минимальная сложность для текущей задачи":
левая часть не платит за интерактивность, которая ей не нужна.
