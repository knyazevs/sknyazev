# ADR-4: Архитектура голосового взаимодействия (ввод/вывод/субтитры/реактивная визуализация)

- Статус: Принято
- Дата: 2026-01-02
- Авторы: Sergey Knyazev, Claude Sonnet 4.6
- Теги: голос, audio, ui, доступность, архитектура, real-time

## Контекст

Правая часть системы является интерактивным представлением: она объясняет и озвучивает ответы, а также визуально реагирует на речь в реальном времени. Голосовой режим — не «фича ради фичи», а способ выразить идею «голос системы», не превращая его в антропоморфного персонажа.

Голосовое взаимодействие должно:

- поддерживать **ввод голосом** (микрофон) и **ввод текстом** (fallback)
- обеспечивать **вывод голосом** (TTS) и **вывод текстом** (субтитры)
- синхронизировать **воспроизведение речи** и **визуальную реакцию** абстрактной сущности
- быть **детерминированным и контролируемым**: UI и голос — представления, не источник истины
- быть **доступным** (можно пользоваться без звука и без микрофона)

Ключевые риски:

- несинхронность голоса/субтитров/визуала
- «дребезг» и нестабильность из-за real-time сигналов
- утечки приватности (аудио), неожиданная запись/хранение
- невоспроизводимые баги из-за гонок состояний
- рост сложности (аудио-пайплайн + UI-анимации + LLM)

## Решение

Ввести архитектуру голосового взаимодействия как три независимых контура, связанные через единый слой состояния (state) и событий (events):

1. **Контур ввода** (Input): микрофон → ASR (опционально) → текст запроса  
2. **Контур вывода** (Output): текст ответа → TTS → аудио воспроизведение  
3. **Контур представления** (Presentation): субтитры + реактивная визуализация, синхронизированные с выводом

### 1) Канонический формат обмена в правой части

Канонический артефакт для дальнейших представлений — это **текст**:
- текст запроса пользователя
- текст ответа системы

Голос и визуальная реакция производны от текста и/или от аудио-сигнала воспроизведения.

### 2) Вывод (TTS) — главный таймлайн для синхронизации

Синхронизацию субтитров и визуализации привязать к **фактическому воспроизведению аудио** (или к его амплитудной огибающей), а не к «ожидаемой длительности текста».

Причина: реальные движки TTS, разная скорость, паузы, буферизация и т.п. делают «оценку длительности» ненадёжной.

### 3) Субтитры — два режима

Поддержать два режима субтитров:

- **Режим A (по предложениям/фразам):** отображение текущей фразы, переключение по событиям вывода (без пометок времени по словам).  
  Это базовый режим, минимально сложный и устойчивый.
- **Режим B (по словам/караоке):** подсветка слова в темпе речи.  
  Включается только если TTS/плеер предоставляет word-level таймкоды либо если есть надёжная аппроксимация (см. ограничения ниже).

По умолчанию принимается режим A, режим B считается опциональным улучшением.

### 4) Реактивная визуализация

Визуальная сущность реагирует на речь через один из источников (в порядке предпочтения):

1. **Анализ реального аудио-выхода** (амплитуда/энергия по окнам времени) из канала воспроизведения TTS  
2. Если доступ к аудио-выходу невозможен: **анимация по “просодии” из текста** (эвристика: паузы/знаки препинания/длина фраз)  
3. Если нет ни того, ни другого: **простая анимация “говорит/молчит”** по событиям старта/окончания воспроизведения

Визуализация не несёт смысловой нагрузки контента, она отображает динамику «голоса системы» (ритм/громкость/паузы), избегая антропоморфизма.

### 5) Модель состояния (conceptual)

Правый интерфейс управляется конечным набором состояний:

- `Idle` — ожидание
- `Listening` — активен микрофон, идёт запись/распознавание
- `Thinking` — обработка/подготовка ответа (контент готовится)
- `Speaking` — воспроизводится TTS
- `Error` — ошибка в любом контуре (с деградацией к тексту)

Переходы инициируются событиями (например: старт записи, получен текст, старт TTS, стоп TTS, отмена, ошибка).

### 6) Деградация и доступность — обязательны

Если отсутствует микрофон/разрешение:
- система полностью работоспособна через текстовый ввод.

Если отсутствует TTS/воспроизведение:
- система показывает текст ответа + субтитры (по сути совпадает), а визуальная сущность работает в режиме “без аудио”.

Если нет звука у пользователя:
- субтитры остаются полноценным каналом.

## Ограничения и правила

1. **Приватность:**
   - аудио пользователя не хранится по умолчанию
   - отсутствует фоновая запись: запись только по явному действию пользователя
   - отображается явный индикатор активного микрофона
2. **Источник истины:**
   - ни распознанная речь, ни TTS не модифицируют публичные данные
   - голосовой слой — представление поверх текста и событий
3. **Управление конкурентностью:**
   - в каждый момент времени активен не более одного “говорящего” источника (один TTS-канал)
   - операции должны быть отменяемыми (stop/cancel): остановка TTS, завершение прослушивания
4. **Стабильность real-time:**
   - визуализация и субтитры обновляются с ограниченной частотой (throttling), чтобы избежать дребезга
   - любые “живые” сигналы проходят сглаживание (smoothing) перед применением к UI
5. **Синхронизация:**
   - если нет надёжной пометкой времени на слова, не симулировать “караоке” как точное — использовать режим A
6. **Ошибки:**
   - сбой в голосовом слое не ломает диалог: система возвращается к тексту и показывает понятное сообщение

## Последствия

### Положительные

- Чётко разделены ответственность и контуры: ввод / вывод / представление.
- Синхронизация опирается на фактическое воспроизведение (меньше рассинхронизаций).
- Доступность и деградация встроены в архитектуру (голос не обязателен).
- Реалтайм-часть изолирована и не влияет на источник истины и контент.

### Отрицательные / компромиссы

- Усложнение: появляется state-machine, управление отменой, буферизация, сглаживание сигналов.
- Караоке-подсветка “по словам” возможна только при наличии корректных таймкодов или дополнительных ухищрений.
- Реакция визуализации без доступа к аудио-выходу будет менее “естественной” (эвристики вместо данных).

## Рассмотренные альтернативы

### Альтернатива A: Только текстовый интерфейс, без голоса
**Отклонено**, потому что:
- теряется центральная идея “голоса системы”
- исчезает демонстрация инженерного контроля над мультимодальностью (аудио + визуал)

При этом текст остаётся обязательным fallback-каналом (см. решение).

### Альтернатива B: Привязка субтитров к оценке длительности текста
**Отклонено**, потому что:
- TTS в реальности меняет скорость, делает паузы, буферизуется
- визуально выглядит как ошибка и снижает доверие

### Альтернатива C: Караоке “по словам” как обязательный режим
**Отклонено**, потому что:
- требует word-level таймкодов от TTS или сложной синхронизации
- увеличивает риск расхождения и добавляет много сложности без критической ценности

### Альтернатива D: Визуализация только по тексту (без анализа аудио)
**Отклонено** как основной подход, потому что:
- хуже отражает реальный ритм/громкость воспроизведения
- меньше ощущение “живого” исполнения

Принято как деградационный режим.

### Альтернатива E: Антропоморфный аватар/персонаж
**Отклонено**, потому что:
- смещает восприятие в сторону “чат-бота/персонажа”
- противоречит идее абстрактной голосовой сущности как “голоса системы”
