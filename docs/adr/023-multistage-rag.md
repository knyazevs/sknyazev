# ADR-23: Многоуровневый RAG-пайплайн

- Статус: Принято
- Дата: 2026-04-14
- Авторы: Sergey Knyazev, Claude Sonnet 4.6
- Теги: rag, retrieval, bm25, hyde, rrf, embeddings, качество

## Контекст

ADR-20 описал базовый RAG: embed(question) → cosine top-5 → LLM.
На практике этот подход даёт плохое качество по четырём причинам:

1. **Потеря контекста при эмбеддинге** — чанк «Архитектура сервиса» без указания
   что это из `adr/014` семантически беден. Модель эмбеддинга не знает, к чему относится секция.

2. **Только косинусная близость** — ADR-документация содержит точные термины:
   названия ADR, технологии, имена. Векторный поиск хуже чем BM25 на точных совпадениях.

3. **Семантический разрыв query–document** — вопрос «какой стек использует Сергей?»
   семантически далёк от текста «Ktor — основной рабочий стек автора».
   Эмбеддинг вопроса ≠ эмбеддинг ответа.

4. **Нет фильтрации по качеству** — top-5 всегда, даже если все 5 нерелевантны.
   LLM получает мусорный контекст и галлюцинирует.

## Решение: трёхслойный retrieval

### Слой 1: Query Reformulation — HyDE

**HyDE (Hypothetical Document Embeddings)**: вместо эмбеддинга вопроса
генерируется гипотетический ответ, который затем эмбеддируется.

```
вопрос → LLM(gpt-4o-mini) → "Сергей использует Kotlin/Ktor..."
                                         ↓
                                   embed(гипотетический ответ)
                                         ↓
                              близко к реальным чанкам документации
```

Причина: вопрос и документ лежат в разных частях векторного пространства.
Гипотетический ответ — в той же области, что релевантные чанки.

Fail-open: при ошибке LLM используется оригинальный вопрос.

### Слой 2: Hybrid Search — Dense + BM25

Параллельно выполняются два независимых поиска:

**Dense (векторный):** `embed(hyde_text) → cosine → top-20`
Хорошо находит семантически близкое.

**BM25 (keyword):** `tokenize(original_question) → BM25 score → top-20`
Хорошо находит точные совпадения: «ADR-19», «Kotlin», «OpenRouter».
Классический BM25: k1=1.5, b=0.75.

Оба источника работают на разных сигналах — дополняют друг друга.

### Слой 3: RRF Fusion + Threshold

**Reciprocal Rank Fusion** объединяет два списка без необходимости нормализации оценок:
```
score(doc) = Σ 1 / (k + rank(doc, list_i)),  k=60
```

После слияния:
- Дедупликация по `entry.id`
- Отсечение по порогу `cosine ≥ 0.35` (чанки ниже порога не попадают в контекст)
- Финальный top-5 для генерации

Если после фильтрации 0 чанков → LLM получает явное указание:
«Релевантной информации в документации не найдено».

### Contextual Chunking

Каждый чанк при эмбеддинге получает breadcrumb-префикс:
```
[adr/014-backend-service.md › Решение › Архитектура сервиса]

...текст чанка...
```

Хранимый контент (`content`) не изменяется — breadcrumb только в embedding-тексте.
Это позволяет модели эмбеддинга учитывать контекст документа.

```
├── HydeService.kt         (NEW) query reformulation via LLM
├── RrfFusion.kt           (NEW) Reciprocal Rank Fusion
└── RagPipeline.kt         (рефактор) оркестрация всех слоёв
```

### Параметры пайплайна

| Параметр | Значение | Обоснование |
|----------|----------|-------------|
| `DENSE_CANDIDATE_POOL` | 20 | Достаточно кандидатов до RRF |
| `BM25_CANDIDATE_POOL` | 20 | Симметрично dense |
| `RRF_K` | 60 | Стандартное значение из оригинальной статьи |
| `SIMILARITY_THRESHOLD` | 0.35 | Отсекает явно нерелевантные чанки |
| `FINAL_TOP_K` | 5 | Баланс контекст/токены для LLM |

## Альтернативы и причины отказа

### Cross-encoder reranking
**Отклонено на старте**: требует hosting отдельной модели (GPU/Cohere API).
Значительно сложнее в деплое. RRF + threshold даёт приемлемое качество без этой сложности.
Легко добавить поверх текущей архитектуры при необходимости.

### Multi-query retrieval
**Отклонено**: генерация N вариантов запроса × N retrieval вызовов = N× latency + N× стоимость.
HyDE решает ту же проблему (semantic gap) дешевле: 1 LLM call вместо N.

### Parent-child chunking
**Отклонено**: усложняет DocumentLoader без значительного выигрыша при текущем
объёме корпуса (~100 чанков). Breadcrumb решает проблему контекста проще.

### Lucene / ElasticSearch для BM25
**Отклонено**: избыточно. Самописный BM25 (~80 строк Kotlin) полностью контролируем,
не требует внешних зависимостей и демонстрирует понимание алгоритма.

## Последствия

### Положительные
- Retrieval качественно улучшается по трём независимым осям
- Точные термины (ADR-номера, технологии) надёжно находятся через BM25
- Семантический разрыв question–document устраняется через HyDE
- LLM никогда не получает нерелевантный контекст (threshold)
- Каждый компонент заменяем независимо

### Отрицательные / Стоимость
- +1 LLM-вызов на запрос (HyDE, ~100ms, gpt-4o-mini)
- Latency первого ответа немного выше
- BM25 не учитывает морфологию русского языка (токенизация по пробелам/пунктуации)
