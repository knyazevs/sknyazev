# ADR-22: Отказ от runtime GitHub API в пользу статической генерации данных

- Статус: Принято
- Дата: 2026-04-22
- Авторы: Sergey Knyazev, Claude Opus 4.7
- Теги: frontend, github-api, build, статика, производительность

## Контекст

Фронтенд (Astro + Svelte) деплоится на GitHub Pages как статика. В проде
браузер напрямую ходил в GitHub API и raw.githubusercontent.com:

- `api.github.com/repos/.../contents/docs` — листинг корня docs
- `api.github.com/repos/.../contents/docs/<dir>` — листинг каждой поддиректории
- `api.github.com/repos/.../git/trees/HEAD:<dir>?recursive=1` — дерево исходников
  для трёх корней (`server/src`, `app/src`, `scripts`)
- `api.github.com/repos/.../commits?per_page=60` — для `/timeline`
- `raw.githubusercontent.com/.../<file>` — содержимое каждого документа и
  каждого исходного файла (включая нагрузочный поиск по коду)

Один полный заход на сайт тратил ≥ 5 запросов к `api.github.com`. Анонимный
лимит GitHub API — 60 запросов в час на IP. На страницах `/code` и при поиске
лимит расходуется ещё быстрее из-за дерева исходников. В результате для
целевой аудитории (работодатели, незалогиненные посетители из одной
корпоративной сети с общим NAT) сайт ломался за минуты — дерево не
загружалось, ADR не открывались, чипы источников от агента вели в 403.

## Решение

Полностью убираем runtime-запросы к GitHub API с фронтенда. Весь контент и
метаданные генерируются на этапе CI-билда как статические файлы и
деплоятся вместе с `app/dist/`.

### Артефакты, генерируемые на билде

- `app/public/data/docs-tree.json` — секции docs/ с лейблами
- `app/public/data/code-tree.json` — дерево исходников из `CODE_SOURCE_DIRS`
- `app/public/data/commits.json` — последние N коммитов из `git log --no-merges`
- `app/public/data/content/<original path>` — побайтовые копии каждого
  документа и исходного файла, включённого в деревья (путь сохраняется
  исходный: `content/docs/adr/001-...md`, `content/server/src/.../X.kt`)

### Поток данных

```
build time:
  scripts/build-static-data.ts
    ├─ читает docs/, server/src/, app/src/, scripts/ (диск)
    ├─ git log (диск)
    └─ пишет JSON + файлы в app/public/data/

astro build:
  app/public/data/  →  app/dist/data/  (штатный копир public/)

runtime:
  browser → fetch('/data/docs-tree.json') и т.п. — same-origin, из CDN GitHub Pages
```

### Общий модуль

Логика построения деревьев, парсинга коммитов и чтения файлов вынесена в
`app/src/lib/docs-data.ts` и используется в двух местах:
- dev-режим: vite-плагин `docs-vite-plugin.ts` отдаёт `/api/docs/*`,
  `/api/code/*`, `/api/commits` — без изменений для разработчика
- prod-билд: `scripts/build-static-data.ts` вызывает те же функции и
  материализует результат на диск

Это гарантирует, что dev и prod читают контент одинаково — расходится
только транспорт.

### Клиент

`app/src/lib/docs-client.ts` оставляет dev-ветку (Vite middleware) без
изменений; prod-ветка теперь строит пути как `${BASE_URL}data/...`. Для
поиска по коду клиент продолжает скачивать файлы пакетами — но уже
с same-origin статики, без лимитов.

### CI

`.github/workflows/deploy.yml`:
- `fetch-depth: 0` на `actions/checkout` — нужен полный git log для `commits.json`
- шаг `bun run build:data` между `build:cv` и `build:app`

## Обоснование решения

- **Нулевые сетевые запросы к GitHub в рантайме.** Лимиты исчезают как класс проблемы.
- **Мгновенная загрузка контента.** Same-origin static вместо 5+ round-trip к api.github.com.
- **Консистентный срез.** Контент соответствует коммиту, с которого собран сайт.
  Невозможен «промежуточный» сбой, когда дерево свежее, а raw-файл ещё не
  доехал до CDN GitHub.
- **Прозрачная работа search.** Поиск по коду теперь скачивает файлы
  с Pages CDN — это быстрее и не съедает чужой лимит.
- **Экономия контекста в CI.** Размер генерируемой data-директории ~800 КБ
  при текущем корпусе docs + исходники. GitHub Pages лимит артефакта — 1 ГБ.

## Последствия

### Позитивные
- Сайт корректно работает из-за любого NAT, из браузеров без refresh-token,
  с любого числа параллельных посетителей.
- Убрана целая категория ошибок «403 rate limit exceeded».
- Производительность `/`, `/code`, `/timeline` вырастает — нет сетевых
  round-trip-ов к внешнему API.

### Негативные / Компромиссы
- Контент обновляется только при деплое. На практике CI срабатывает на
  `push: main` в течение минут, что приемлемо для неоперативного сайта-портфолио.
- Небольшой рост размера статического артефакта (на ~800 КБ сейчас,
  линейно от объёма `docs/` и `CODE_SOURCE_DIRS`). Не ограничивает деплой.
- Build-скрипт импортирует из `app/src/lib/docs-data.ts`, который
  импортирует `app/src/config.ts`. Пришлось обезопасить доступ к
  `import.meta.env` в `config.ts`, чтобы модуль грузился в Node без Vite.

## Рассмотренные альтернативы

- **GitHub token на клиенте (5000 запросов/час).** Отклонено: токен
  неизбежно попадает в JS-бандл, виден любому посетителю. Украденный
  токен выжигает лимит владельца за минуты.
- **Прокси на собственном Ktor-бэкенде с кешем.** Технически работает,
  но создаёт зависимость фронта от бэка для полностью статического
  контента, нагружает сервер, который сейчас занят RAG, и добавляет
  новую точку отказа.
- **Conditional requests (ETag) к GitHub API.** Не помогает: 304-ответы
  тоже списываются из rate-limit-пула для анонимных запросов.
- **Hybrid: загрузка на билде + обновление из GitHub API как fallback.**
  Отклонено как преждевременная сложность — статика сама по себе
  решает задачу, fallback не нужен.

## Примечание

ADR-012 (docs-reading-strategy) фиксирует старую стратегию чтения
с `api.github.com` + `raw.githubusercontent.com`. Данный ADR заменяет
её для prod-пути. Dev-пайплайн (Vite middleware) сохранён как есть.

## Связанные артефакты

- `scripts/build-static-data.ts` — генератор
- `app/src/lib/docs-data.ts` — общая логика
- `app/src/lib/docs-client.ts` — клиентская часть (prod читает `/data/...`)
- `.github/workflows/deploy.yml` — шаг `build:data` в CI
- `docs/blog/2026-04-22-github-api-rate-limit.md` — живой нарратив
