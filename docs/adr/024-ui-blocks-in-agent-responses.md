# ADR-24: UI-блоки в ответах агента

- Статус: Принято
- Дата: 2026-04-22
- Авторы: Sergey Knyazev, Claude Opus 4.7
- Теги: ui, blocks, rag, llm, tools, sse, frontend

## Контекст

Сейчас ответ агента — plain-text streaming плюс массив `sources` в конце (ADR-016, ADR-019). Визуально ответ одномерный: текст и чипы источников под ним. Для CV-сайта, цель которого — продемонстрировать автора работодателю, этого недостаточно. На вопрос «кто ты?» хочется показать фото. На вопрос «над какими проектами работал?» — карточки с изображениями и ссылками, а не только пути к документам.

Задача: дать агенту возможность вставлять в ответ богатые визуальные блоки (изображения, галереи, списки ссылок), сохранив streaming-UX и совместимость с fallback-цепочкой моделей (ADR-023), часть которых не поддерживает tool-calling.

## Решение

### Архитектура из двух независимых механизмов

Блоки порождаются из двух источников одновременно:

**C — автоизвлечение блоков из markdown при RAG-ретривале.**
Когда RAG находит чанк, соответствующий части документа, бэкенд парсит markdown-тело этого чанка и автоматически превращает найденные `![alt](url)` и `[title](url)` в соответствующие UI-блоки.

**B — генерация блоков LLM через OpenRouter tools.**
LLM в system prompt получает каталог «render-инструментов» (`render_image`, `render_image_gallery`, `render_link_list`) и вызывает их, когда считает визуальную подачу уместной. Бэкенд парсит tool_calls из SSE-стрима OpenRouter и эмитит блоки сразу.

Механизмы независимы. Используют общий тип `UiBlock` и общий SSE-канал. Работают параллельно в одном запросе: C включён всегда, B — только на моделях с tool-support.

### Блоки как домен-агностичные UI-примитивы

Блок — не доменная карточка («профиль», «проект», «ADR»), а атомарный визуальный примитив. Композицию собирают или LLM, или C-диспетчер, или автор контента через markdown.

MVP-каталог:

| Тип | Поля | Применение |
|-----|------|-----------|
| `image` | `{url, caption?, alt?, link?}` | Одна картинка |
| `image_gallery` | `{images: [{url, caption?}]}` | Сетка из 2+ картинок |
| `link_list` | `{items: [{url, title, description?}]}` | Внешние ресурсы, демо, репозитории |

Каталог расширяемый. Добавление нового блока = новый Kotlin data class в `UiBlock.kt` + Svelte-компонент + регистрация в `BlockRegistry`. Ядро диспетчера и протокол SSE не меняются.

### Механизм C: markdown → blocks

`DocumentLoader` при чанкинге `.md` файлов дополнительно парсит тело каждого чанка:

- `![alt](url)` → `image`
- 2+ последовательных `![]()` в одном чанке → один `image_gallery`
- внешние и относительные `[title](url)` без якорей (`#section` игнорируются) → накапливаются в один `link_list`

Результат сохраняется в новом поле `DocumentChunk.blocks: List<UiBlock>`, рядом с `content`, `breadcrumb`, `filePath`.

**Путь изображений.** Автор пишет в markdown относительный путь рядом с файлом:

```markdown
![Сергей](profile.png)
```

Билд-скрипт `scripts/build-static-data.ts` при копировании `docs/` в `app/public/data/content/` дополнительно:

1. Сканирует каждый `.md` на `![](path)` и `[](path)` с относительным путём.
2. Копирует файл `docs/{doc-dir}/{path}` в `app/public/images/{doc-rel-path}/{path}`.
3. В копии markdown переписывает путь на абсолютный `/images/{doc-rel-path}/{path}`.

URL вида `http(s)://...` не трогаются — используются как есть. Якорные ссылки `#section` игнорируются на уровне парсера блоков (но остаются в тексте как обычный markdown).

**Дедупликация текста и блоков.** Если markdown-строка превратилась в блок, соответствующая строка **удаляется** из `content`, который уходит в LLM-контекст. Иначе LLM процитирует картинку или ссылку в тексте, и они появятся дважды: один раз — блоком, другой — через `marked.parse()` во фронте.

**Диспетчер.** В `RagPipeline.kt` после RRF-фильтрации и формирования `finalChunks` появляется новый шаг: `BlockDispatcher.dispatch(finalChunks)` собирает блоки всех финальных чанков в один список и передаёт в SSE-канал для эмита.

### Механизм B: OpenRouter tools

`OpenRouterClient.streamChat` получает опциональный параметр `tools: List<ToolDefinition>`. Если передан — в JSON-запрос добавляются `"tools": [...]` и `"tool_choice": "auto"`. Если модель не поддерживает tools (per-модель capability) — `tools` не отправляется, B-часть на этой модели отключена.

SSE-парсер теперь обрабатывает два вида дельт:
- `delta.content` → токен в `Flow<ChatEvent.Token>` (как раньше)
- `delta.tool_calls` → фрагмент, накапливаемый по `index` до полного JSON аргументов, затем эмит `Flow<ChatEvent.ToolCall>`

ChatRoutes.kt подписан на оба — и пробрасывает токены как `token`-события, а tool_calls передаёт в `UiTools.execute(name, args)`, который валидирует аргументы, строит `UiBlock` и эмитит его в тот же SSE-канал.

**Каталог tools в MVP:**

- `render_image(url: String, caption?: String, alt?: String, link?: String)`
- `render_image_gallery(images: List<{url: String, caption?: String}>)`
- `render_link_list(items: List<{url: String, title: String, description?: String}>)`

Handler каждого tool — тонкий: валидация (URL-формат, max length полей) → `UiBlock` → эмит.

**System prompt** при включённом B добавляет секцию:
1. Когда уместно — используй tools для визуальной подачи.
2. Если передал ресурс через tool — не повторяй его в markdown-тексте.
3. Всего блоков в ответе не больше 5.

### Инвариант fallback-цепочки

ADR-23 устанавливает: fallback-модель пробуется только до первого эмита. Этот инвариант **сохраняется**. Tool-поддержка проверяется per-модель до первого запроса: если primary поддерживает tools и первый запрос ушёл с `tools` — fallback-модель тоже получит запрос с `tools` (если она их поддерживает) или без (если не поддерживает). Инвариант согласованности ответа не нарушается: каждая модель видит один консистентный запрос.

### SSE-протокол

К существующим событиям (`token`, `sources`, `suggestions`, `error`, `[DONE]`) добавляется четвёртое:

```
data: {"block": {"type": "image", "data": {"url": "/images/profile/profile.png", "caption": "Сергей"}}}
```

Множество таких событий стримятся по мере появления: C эмитит сразу после RRF, до начала генерации текста; B — по мере срабатывания tool_calls (вплетённых моделью в content-поток). Порядок в SSE = порядок появления, фронт сохраняет его в массив `extraBlocks`.

**Позиционирование.** Все блоки рендерятся в единой зоне `div.msg-blocks` между текстом ответа и `div.msg-sources` (чипами источников). Inline-блоки (встроенные в поток токенов) — не в MVP.

### Ограничения

- **Hard cap: максимум 5 блоков на ответ суммарно C+B.** При превышении лишние дропаются на бэкенде с WARN-логом.
- **Дубликаты между C и B не фильтруются в MVP.** Если C эмитнул картинку из контекста, а B вызвала `render_image` с тем же URL — показываются обе. Фильтр отложен до появления реальной проблемы.
- **Только `after`-позиция.** Inline-блоки откладываются.

### Frontend

- `BlockRegistry.ts` — карта `type → Svelte component`, центральная точка регистрации.
- Новые компоненты в `app/src/components/blocks/`: `ImageBlock.svelte`, `ImageGalleryBlock.svelte`, `LinkListBlock.svelte`.
- `RightPanel.svelte` — новое состояние `extraBlocks: UiBlock[]` на сообщение ассистента, парсер SSE-события `block`, рендеринг блоков через `BlockRegistry` в `.msg-blocks` между текстом и `.msg-sources`.
- Санитайзер финального markdown перед `marked.parse()`: удаляет `![](...)` и внешние `[](...)` — страховка на случай, если LLM не послушала system prompt.

## Обоснование

- **Markdown как источник правды.** Автор не учит новый DSL и не дублирует данные в frontmatter — пишет обычный markdown, блоки появляются сами. DRY соблюдён.
- **Домен-агностичные примитивы.** Новый тип контента (условные «отзывы клиентов», «дипломы») не требует нового типа блока — используется `image`, `link_list` или будущий `quote`. Каталог блоков растёт медленно.
- **C + B покрывают разные сценарии.** C отрабатывает «тривиальную визуализацию контента», которая уже есть в документах (фото из профиля, скриншоты из `experience/`). B — «умный UX», где LLM компонует ответ из примитивов под конкретный вопрос («покажи мои GitHub-репо» → `render_link_list` с отобранными ссылками).
- **Совместимость с fallback.** C работает на любой модели и является базовой линией. B отключается на моделях без tool-support, ответ деградирует плавно: блоки из markdown остаются, tool-блоки пропадают.
- **Streaming сохраняется.** Блоки стримятся отдельными SSE-событиями и не блокируют текстовый поток. Пользователь видит текст сразу, блоки — по мере готовности.

## Последствия

### Положительные
- Ответ визуально богаче: CV-сайт превращается из «чат с документами» в «живое представление автора».
- Автор публикует изображения и ссылки в markdown привычно, как в обычной статье — нет отдельной UI-прослойки.
- Архитектура расширяется через регистрацию новых блоков, ядро не трогается.

### Отрицательные / Компромиссы
- SSE-протокол усложняется (четвёртый тип события), парсеры на обеих сторонах добавляют кода.
- Требуется наполнение контента: сейчас в `docs/` изображений нет вообще, кроме `profile/profile.png`. Без картинок и внешних ссылок в markdown C ничего не покажет — нужен отдельный раунд работы над content.
- `OpenRouterClient` становится чувствителен к capability-модели (добавляется матрица: поддерживает ли модель tools).
- Дубликаты C и B в MVP допустимы — UX может иногда показать картинку дважды, пока не добавим фильтр.
- Hard cap в 5 блоков — эвристика. При реальном использовании может потребоваться калибровка.

## Рассмотренные альтернативы

### Только C (автоизвлечение из markdown)
Отклонено: LLM не может инициативно показать блок, которого нет в ретривале. Например, собрать сравнительный `link_list` по проектам, каждый из которых представлен разным чанком.

### Только B (tools)
Отклонено: для простых кейсов («фото автора из профиля, который и так попал в RAG») LLM должна явно вызывать tool — лишние токены, лишняя когнитивная нагрузка. C эти кейсы закрывает бесплатно.

### MCP-сервер для UI-блоков
Отклонено: MCP стандартизирует обмен *внешней* модели с *внешним* сервером инструментов (Claude Desktop ↔ сервер). В нашем пайплайне и модель, и бэкенд, и фронт — одно приложение. MCP добавил бы сетевой хоп и зависимость без ценности.

### Frontmatter как источник блоков
Отклонено: автор должен был бы дублировать в frontmatter то, что уже присутствует в теле markdown. Нарушение DRY. Плюс невозможно описать «всё что есть в документе» декларативно — markdown всё равно нужно парсить.

### Статические «консервы» (`app/public/data/blocks/*.json`)
Отклонено: требуют отдельного источника данных (руками или автогенерацией из чего-то). Автогенерация из markdown — и есть механизм C, который мы принимаем. Runtime-параметризация B-tools несовместима со статикой.

### Inline-блоки (встроенные в текст ответа)
Отложено: сложность парсинга, разметки позиций и UX-сглаживания не оправдана в MVP. Все блоки рендерятся `after`. Вернуться к вопросу, когда появится конкретный UX, который без inline невозможен.

### Один SSE-ивент со всеми блоками в конце
Отклонено: теряется streaming-преимущество B. Блоки «вспрыгивают» только после `[DONE]`, хотя готовы раньше.
