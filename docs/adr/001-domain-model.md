# ADR-1: Доменная модель персонального архитектурного профиля

- Статус: Принято
- Дата: 2026-01-02
- Авторы: Sergey Knyazev, LLM (ChatGPT 5.2), Claude Sonnet 4.6
- Теги: процесс, управление, документация

## Контекст

Данный проект является персональным архитектурным профилем человека —
инженера и технического лидера.

Цель проекта — показать **мышление, подход и решения**, которые человек
применяет при проектировании и развитии сложных систем,
не превращая самого человека в продукт или формальную систему.

Для этого необходимо отделить:
- личность автора
- его инженерные принципы и решения
- результаты применения этих решений

Без явной доменной модели:
- архитектурные решения теряют связность
- становится невозможным корректное ограничение LLM
- разные представления начинают расходиться
- проект перестаёт быть управляемым как инженерный артефакт

---

## Решение

Определить доменную модель, в которой:

- **человек остаётся центральным субъектом**
- инженерные решения, принципы и артефакты являются объектами описания
- опыт представлен через решения и их последствия, а не через абстрактные роли

Доменная модель служит:
- источником структурированных данных
- основой для архитектурной навигации
- контрактом между документацией, UI и LLM
- способом сохранить инженерную строгость без обезличивания автора

---

## Границы домена

### В домен входят:
- инженерные и архитектурные решения
- принципы и подходы, применяемые автором
- системы, над которыми велась работа
- ограничения и контекст, влияющие на решения
- последствия и наблюдаемые результаты

### Из домена исключены:
- оценочные суждения о личности
- маркетинговые формулировки
- неподтверждённые утверждения
- абстрактные soft skills без инженерного контекста
- гипотетический или вымышленный опыт
- названия компаний и клиентов (NDA): системы описываются через тип, масштаб и контекст

---

## Ключевые доменные сущности

### Person
**Описание:** Человек — автор архитектурных решений  
**Примечание:** Не является системой или продуктом

- id
- name
- professional-focus
- summary

---

### System
**Описание:** Реальная система или платформа, над которой велась работа

Названия компаний не используются (NDA). Вместо этого — описательный ярлык,
отражающий тип и масштаб системы.

Пример: `"Платёжная платформа, B2C, ~2M транзакций/день, миграция с монолита"`

- id
- name (описательный ярлык: тип + домен + масштаб, без названия компании)
- context (организационный контекст: стартап / scaleup / enterprise, размер команды)
- lifecycle-stage (greenfield / scaling / legacy-migration / sunset)
- author-role (decision-maker / influencer / builder / mixed)
- constraints
- outcomes

---

### Principle
**Описание:** Инженерный принцип, которым руководствуется автор

- id
- name
- description
- motivation
- examples
- related-decisions

---

### Decision
**Описание:** Конкретное архитектурное решение, принятое автором

- id
- title
- context
- decision
- rationale
- consequences
- alternatives
- related-systems

---

### Constraint
**Описание:** Ограничение, влияющее на принятие решений

- id
- type (technical / organizational / business)
- description
- impact

---

### TradeOff
**Описание:** Осознанный компромисс между вариантами

- id
- options
- chosen-option
- rationale
- cost

---

### Outcome
**Описание:** Наблюдаемый результат принятого решения

- id
- description
- metrics (если применимо)
- related-system

---

### Artifact
**Описание:** Материальный результат инженерной работы

- id
- type (code / diagram / ADR / PoC)
- location
- related-decisions

---

## Связи между сущностями

```text
Person
 ├─ applies → Principle
 ├─ makes → Decision
 └─ works-on → System

Decision
 ├─ constrained-by → Constraint
 ├─ involves → TradeOff
 ├─ affects → System
 └─ leads-to → Outcome

System
 ├─ embodies → Principle
 ├─ contains → Artifact
 └─ exhibits → Outcome
```

---

## Источник истины

Источником истины являются:

* зафиксированные архитектурные решения (ADR)
* подтверждённые факты опыта
* доменные артефакты проекта

Ни UI, ни LLM, ни визуальные представления
**не могут изменять доменные данные**.

---

## Роль доменной модели в проекте

Доменная модель используется:

* для структурирования архитектурного профиля
* как основа для всех представлений
* как входные данные для LLM-интерпретации
* как средство поддержания целостности проекта

Любое новое представление обязано
следовать данной модели.

---

## Последствия (Consequences)

### Положительные

* Чёткое разделение личности и решений
* Управляемая сложность
* Возможность эволюции проекта
* Контролируемая интерпретация опыта

### Отрицательные

* Повышенная формальность
* Необходимость дисциплины в поддержке
* Более медленный старт

Данные издержки считаются осознанными и приемлемыми.

---

## Примечание

Доменная модель описывает **не человека как систему**,
а инженерные решения и подходы,
которые человек применяет в реальных проектах.
