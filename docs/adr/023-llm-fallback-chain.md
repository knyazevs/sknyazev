# ADR-23: Fallback-цепочка LLM при сбое primary-модели

- Статус: Принято
- Дата: 2026-04-22
- Авторы: Sergey Knyazev, Claude Opus 4.7
- Теги: llm, openrouter, отказоустойчивость, deploy, конфигурация

## Контекст

Сервис шлёт стриминговые запросы в OpenRouter через `OpenRouterClient.streamChat`.
При этом есть два типа отказов, которые OpenRouter может вернуть незаметно:

1. **Явный не-2xx** — неверный ключ, превышен кредит, модель не существует,
   rate limit. HTTP-клиент Ktor по умолчанию не бросает на такие статусы,
   а просто возвращает ответ; SSE-парсер в этом случае видит не-`data:`
   лины и тихо выходит без эмитов.

2. **HTTP 200 с нулём content-токенов.** Самый коварный кейс. Встречается,
   когда:
   - имя модели указано неверно (например, deprecated id или опечатка),
   - модель вернула только reasoning-чанки без итогового `delta.content`,
   - модель отфильтровала запрос по safety-политике.

Оба сценария приводили к пустому `Flow<String>`: frontend получал только
`sources`, `suggestions` и `[DONE]`, без текста ответа. Локальный dev
работал (правильная модель в `.env`), prod молчал (опечатка в
`LLM_MODEL` в GitHub Secrets). Диагностировать можно было только по логам
сервера, и то если смотреть в нужный момент.

Кроме того, фронт не видел осмысленной ошибки — просто «ответ пустой» без
причины. Пользователь делал вывод «бот сломался».

## Решение

### Структура

`OpenRouterClient.streamChat` принимает `primaryModel` + `fallbackModels: List<String>`.
Попытки выполняются последовательно: primary → fallback[0] → fallback[1] → …

Отдельный приватный метод `streamFromModel(modelName, messages)` делает один
попытку — отправляет запрос, парсит SSE, эмитит токены. На двух типах сбоя
бросает `LlmEarlyFailure`:

- HTTP-ответ не 2xx (тело первые 500 байт включены в сообщение для диагностики)
- HTTP 200, но после полного прочтения стрима не эмитнут ни один content-токен

Внешний `streamChat` ловит только `LlmEarlyFailure` и идёт к следующей модели
в списке. Любое другое исключение (mid-stream cancel, сетевой обрыв после
первых эмитов) пробрасывается как есть — fallback не пытается.

### Инвариант безопасности

**Fallback возможен только до первого эмита.** Это критически важно:
если мы начали отдавать токены во фронт и упали в середине, переключение на
другую модель даст пользователю склейку двух разных ответов — хуже, чем
просто ошибка. Инвариант поддерживается тем, что `LlmEarlyFailure`
бросается только в ветках «статус не 2xx» (до открытия стрима) и «emitted == 0
после завершения стрима» (т.е. никакой `emit(token)` не сработал).

### Конфигурация

Новое поле в `AppConfig`: `llmFallbackModels: List<String>`.
Читается из env-переменной `LLM_MODEL_FALLBACKS` (comma-separated, trim,
пустые элементы отфильтровываются). Пусто → fallback отключён, поведение
эквивалентно старому.

Дефолт в `application.yaml`:
```yaml
fallbackModels: "$LLM_MODEL_FALLBACKS:anthropic/claude-sonnet-4-6,openai/gpt-4o"
```

То есть из коробки: primary Opus 4.7 → Sonnet 4.6 → GPT-4o. Смешанный
провайдер-mix в дефолте специально — если упал весь Anthropic, gpt-4o всё
ещё ответит.

### Видимость

В `Bootstrap.kt` при старте пишется строчка:
```
LLM fallback chain: anthropic/claude-opus-4-7 → anthropic/claude-sonnet-4-6 → openai/gpt-4o
```

Каждая неудачная попытка даёт `WARN` с причиной:
```
LLM fallback: 'anthropic/claude-opus-4-7' failed (1/3) — ... status=404 ... "No allowed providers for this model"
```

Если упали все — `error()` с конкатенацией причин: `ChatRoutes.catch`
подхватывает это как обычную ошибку и эмитит `data: {"error": "..."}`
в SSE. Frontend уже умеет рендерить этот ивент.

## Обоснование решения

- **Robustness против человеческой ошибки.** Опечатка в имени модели
  в GitHub Secrets теперь не убивает сервис — Sonnet или GPT-4o подхватят.
- **Видимость.** Переход на fallback логируется, администратор видит,
  что primary проблемна, и может починить именно её.
- **Отсутствие побочных эффектов в happy path.** Если primary работает —
  fallback'и никогда не пробуются, дополнительной латентности нет.
- **Кросс-провайдер.** В дефолте Anthropic + OpenAI. Если отвалится весь
  Anthropic-провайдинг у OpenRouter (бывало), GPT-4o ответит.

## Последствия

### Позитивные
- Устраняется класс отказов «тихий пустой ответ».
- Ошибка «все модели упали» видна пользователю, а не маскируется под
  пустой ответ.
- Диагностика резко улучшается: в логе видно конкретную причину от OpenRouter.

### Негативные / Компромиссы
- При реальном отказе primary пользователь получает ответ **с задержкой**
  на один прокси-round-trip до OpenRouter. Практически это ~300-800ms.
  Приемлемо.
- Fallback-модели могут отвечать в другом стиле/длине/качестве, чем primary.
  Читатель этого не замечает, потому что ответ всё равно один, но стиль
  сайта при вырожденной prod-конфигурации может колебаться.
- Если настроить цепочку плохо (все fallback'ы тоже невалидны), latency
  при полном отказе = сумма таймаутов всех моделей. Решается кратким
  списком (2-3 модели) и разумным `requestTimeoutMillis` (сейчас 60с).

## Рассмотренные альтернативы

- **Retry той же модели.** Отклонено: если модель неверно названа или
  закончились кредиты, повтор не поможет. Retry имеет смысл только на
  транзиентные сетевые ошибки, которых у нас считанные единицы в месяц.
- **Fallback в середине стрима.** Отклонено — приводит к склейке двух
  ответов, хуже чем одна ошибка.
- **Circuit breaker на primary.** Отклонено как преждевременное. При
  нашем трафике отдельный breaker с таймерами/состоянием не даёт ценности:
  если primary отвалилась на день, мы всё равно на каждый запрос будем
  пробовать её и фолбэчить — это не дорого.
- **Буферизовать primary-ответ до конца, решать «принять или перейти»
  по итогам.** Отклонено: ломает стриминговый UX, добавляет память.
- **Валидировать модель на старте (список OpenRouter-моделей).** Не
  решает сценарий с исчерпанным кредитом и моделями, которые внезапно
  стали недоступны после старта.

## Примечание

ADR-016 (RAG-пайплайн) описывает общую архитектуру вызова LLM.
Данный ADR уточняет поведение в сбойных сценариях и не меняет базовый
пайплайн. Ограничение «fallback только до первого эмита» — часть
инварианта согласованности ответа, этот инвариант должен сохраняться
при любых дальнейших правках транспорта.
