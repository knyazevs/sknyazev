# Сайт, который ломался после шестого клика

- Дата: 2026-04-22
- Теги: github-api, rate-limit, статика, билд, урок

## Контекст

Фронтенд проекта — это Astro + Svelte на GitHub Pages, голая статика.
Чтобы не дублировать контент в репозиторий дважды (отдельно в `docs/`,
отдельно в бандле), я сделал хитро: в проде браузер ходит напрямую в
`api.github.com` и `raw.githubusercontent.com`, тянет дерево `docs/`,
дерево исходников, список коммитов и содержимое файлов. Один репозиторий
— один источник истины. Красиво.

## Что думал изначально

Эта схема казалась мне элегантной. Я помнил, что у GitHub API есть
какой-то rate limit, но подсознательно списал это на «60 запросов в
минуту» — думал, что минута это много. Дерево docs/ маленькое, десяток
ADR, запросов хватит. Решил — разберусь, если станет проблемой.

## Что пошло не так

Проверил поведение сайта с «холодным» клиентом — всё быстро. Открыл
через пару минут из второго браузера — уже медленнее. На третьем заходе
дерево просто не загрузилось: API отдал 403.

Полез в доку. Лимит — **60 запросов в час на IP**, не в минуту. И это
не на пользователя — это на NAT-шлюз. Одна корпоративная сеть ~
одна квота. А я на каждой загрузке главной трачу минимум пять запросов
только чтобы собрать дерево `docs/`: корень плюс четыре-пять
поддиректорий. `/code` открывается — ещё три запроса `git/trees` по
исходным директориям. `/timeline` — ещё один на коммиты. Итого
десяток запросов за переход по трём страницам. Двенадцать
посетителей из офисной сети — и сайт мёртв до следующего часа.

Хуже всего, что целевой читатель этого сайта — работодатель. Как раз
из корпоративной сети с общим NAT, через который уже до него сходило
десять коллег. Он открывает CV-проект — а там пустое дерево и 403
в консоли. Демонстрация «архитектурного мышления» провалена на уровне
сетевого слоя.

## Решение и вывод

Убрал runtime-ходы в GitHub API полностью. Теперь на CI в процессе
билда отдельный скрипт читает `docs/`, исходники и `git log` с диска
и материализует всё в `app/public/data/`:

- `docs-tree.json`, `code-tree.json`, `commits.json`
- `content/<original path>` — побайтовая копия каждого документа и
  исходника, с сохранением путей

Фронт просто делает `fetch('/data/...')` — same-origin, GitHub Pages
CDN, никакого rate-limit. Весь артефакт — ~800 КБ, копеечно.

Чтобы не дублировать код между dev (Vite middleware) и prod (билд-скрипт),
вынес всю логику построения деревьев и чтения файлов в `app/src/lib/docs-data.ts`.
Vite-плагин и скрипт билда теперь вызывают одни и те же функции — просто
в одном случае это HTTP-хендлер, в другом — запись в файл.

Как бонус:
- сайт загружается заметно быстрее — нет пяти round-trip к внешнему API
- поиск по коду не долбит `raw.githubusercontent.com` с каждого посещения
- контент всегда совпадает со снапшотом репозитория, с которого собрали
  сайт — исчезает класс багов «дерево свежее, а файл ещё не в CDN»

## Что бы сделал иначе

Главный урок — **разовая проверка rate-limit на голой документации до
архитектурного решения, а не после первого инцидента**. Пять минут в
`curl https://api.github.com/rate_limit` без авторизации показали бы
60/hour ещё до того, как я положил runtime-зависимость на внешний API
в статический сайт.

Второй урок — static site должен быть по-настоящему static. Если я
уже плачу за то, что всё деплоится как файлы, нет никакого смысла
потом ходить куда-то за данными, которые были на диске во время билда.
Это нарушение принципа «тот, кто собрал сайт, уже имел весь контент на
руках». Нужно было сразу сгенерировать всё на билде и забыть.

И третий — соблазн «использовать GitHub API как бесплатный CMS»
возвращается каждый раз, когда сайт живёт в одном репозитории с
контентом. Теперь знаю: в проде этот соблазн стоит ошибок с
диагностикой «403 после шестого клика».
